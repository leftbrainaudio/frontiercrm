    ADR-001: Backend Framework — FastAPI

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: AI/ML ecosystem for conversation intelligence pipeline

    Context

    FrontierCRM needs a backend framework that handles: - CRUD-heavy REST API with complex domain models (contacts, deals, pipelines) - Real-time WebSocket connections (pipeline updates, notifications) - Async background processing (email sync, calendar sync, voice transcription) - AI/ML pipeline integration (Whisper, LLM inference for conversation intelligence) - Strong type safety for a data-intensive domain - Team productivity (3–5 engineers shipping in 3 months)

    Options Considered

    Framework Language Async WebSocket AI/ML Ecosystem Type Safety Hiring
    FastAPI Python ✅ Native ✅ Via WS ✅ Best (HF, Torch, Whisper) ✅ Pydantic ✅ Large
    Django + DRF Python ⚠️ Async added later ⚠️ Channels ✅ Same as FastAPI ✅ Django ORM ✅ Largest
    NestJS TypeScript ✅ Native ✅ Native ⚠️ Via Python subprocess ✅ TypeScript ✅ Large
    Gin/Echo Go ✅ Goroutines ✅ Native ❌ No native ML libs ⚠️ Interface-based ⚠️ Smaller
    Phoenix Elixir ✅ Native ✅ GenServer ❌ No native ML libs ✅ Pattern match ❌ Niche

    Decision

    Python (FastAPI) — selected.

    The deciding factor is the AI/ML pipeline. FrontierCRM's primary differentiator is conversation intelligence. Running that in a separate Python microservice adds deployment complexity, serialization overhead, and a second codebase to maintain. FastAPI lets the entire stack live in one language with one async runtime.

    Consequences

    • Positive: Single language for backend + AI/ML pipeline. Pydantic validation for CRM domain. Auto-generated OpenAPI docs. Excellent hiring pool.
    • Negative: Lower raw throughput than Go/Node (~7,000 req/s vs ~80,000). Must use SQLAlchemy async sessions carefully.
    • Mitigation: Voice processing runs on separate Celery workers, not in the HTTP hot path. Bottlenecks are external API rate limits (Gmail, Graph), not our response time.

    ADR-002: API Design — REST-first

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Webhooks, caching, public API simplicity

    Context

    FrontierCRM's API serves four clients: the React SPA, the PWA mobile app, third-party integrations via a public API, and webhook consumers. The CRM domain is heavily CRUD with complex query patterns.

    Options Considered

    Paradigm Complexity Caching Webhooks Rate Limiting Client Flexibility
    REST Low ✅ Easy (HTTP cache) ✅ Native ✅ Per-endpoint Low
    GraphQL Medium ❌ Hard (no HTTP caching) ⚠️ Extra tooling ❌ Per-field Medium
    REST + selective GraphQL High Mixed Mixed Complex High
    gRPC High ❌ No browser ❌ No browser ✅ Easy ❌ Browser requires gRPC-web

    Decision

    REST-first — selected.

    Conventions: - JSON:API-inspired compound documents with ?include= - Cursor-based pagination (?cursor=...&limit=50) for lists - Filtering: ?filter[status]=won&filter[owner_id]=42 - Sorting: ?sort=-created_at (minus prefix = descending) - Sparse fieldsets: ?fields[deal]=title,value,stage - Versioning: /api/v1/... URL prefix - Idempotency key header for mutation endpoints - Rate limiting: per-tenant, per-endpoint, sliding window via Redis

    Consequences

    • Positive: Instantly comprehensible to any REST developer. Excellent caching story. Native webhook semantics.
    • Negative: Frontend may need multiple roundtrips for complex dashboards — mitigated by compound document responses and client-side caching with TanStack Query.
    • Neutral: Adding GraphQL later is possible as a separate /graphql endpoint without breaking REST.

    ADR-003: Database — PostgreSQL + Redis + TimescaleDB

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: RLS multi-tenancy, JSONB custom fields, analytics via extension

    Context

    Highly relational data (contacts→deals→activities→users) with multi-tenant isolation, custom fields, full-text search, time-series analytics, and geo-location.

    Decisions

    Component Choice Rationale
    Core Database PostgreSQL 16 RLS for multi-tenancy, JSONB for custom fields, tsv for basic search
    Analytics TimescaleDB (pg extension) Hypertables on existing PG — no new infra
    Cache/Queue/PubSub Redis 7 Single service for three roles
    Multi-tenancy Row-Level Security Simple ops, good isolation for SMB data
    Custom Fields JSONB column Flexible, indexable, simple queries. Not EAV pattern.

    Consequences

    • Positive: Single DB for all workloads. No separate analytics infrastructure. RLS provides tenant isolation without schema-per-tenant complexity.
    • Negative: RLS can cause subtle bugs if app.current_tenant_id is not set — mitigated by required middleware that raises 500 if missing.
    • Negative: Redis must be configured with persistence (AOF + RDB) — losing Redis = losing queued tasks.

    ADR-004: Frontend — React 19 + TypeScript + Next.js 15

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Largest ecosystem, TanStack/dnd-kit for complex CRM UI

    Context

    Complex data views (Kanban board, data tables, nested detail panels), rich interactivity (drag-and-drop, inline editing), real-time updates, responsive design, WCAG 2.1 AA accessibility, SSR for marketing pages.

    Options Considered

    Framework Bundle Size Performance Ecosystem SSR Hiring
    React 19 + Next.js 15 ~80KB gzip Good (React compiler) ✅ Largest ✅ Next.js ✅ Largest
    Vue 3 + Nuxt ~50KB gzip Good ✅ Large ✅ Nuxt ✅ Large
    Svelte 5 + SvelteKit ~25KB gzip Excellent ⚠️ Growing ✅ SvelteKit ⚠️ Smaller
    Solid.js + SolidStart ~15KB gzip Excellent ⚠️ Small ⚠️ Experimental ❌ Niche

    Decision

    React 19 + TypeScript + Next.js 15 — selected.

    Key libraries: - TanStack Query — server state management, caching, optimistic updates - TanStack Table — sortable, filterable, paginated data tables - dnd-kit — drag-and-drop kanban pipeline boards - Radix UI + shadcn/ui — accessible primitive components - Tailwind CSS 4 — utility-first styling - React Email — email templates shared with Resend - zod — runtime validation matching Pydantic models

    Consequences

    • Positive: Largest ecosystem for complex CRM UI (data grids, kanban boards, rich text). Radix provides accessible primitives out of the box. TypeScript improves maintainability.
    • Negative: Bundle size (~80KB gzip) larger than alternatives — mitigated by Next.js route-level code splitting and React Server Components.

    ADR-005: Mobile — PWA-first, React Native deferred

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Zero app store friction, single codebase, SMB mobile needs

    Context

    Sales teams need to check pipelines, log activities, and view contacts from mobile. SMB segment includes deskless workers in field sales.

    Options Considered

    Approach Reuse Web Code Native Feel Dev Cost Maintenance
    PWA (React responsive) ✅ 100% ⚠️ Web view Low Single codebase
    React Native ⚠️ 60–70% ✅ Near-native Medium-High Two codebases
    Flutter ❌ 0% ✅ Native-quality High Two codebases
    Hybrid (Capacitor) ✅ 100% ❌ WebView UX Medium Single codebase

    Decision

    PWA-first — selected for Phase 0–2. Evaluate React Native in Phase 4 when ARR > $500K and mobile usage analytics show >30% mobile sessions.

    Phased approach: - P0–P1: Responsive web (desktop + tablet + phone) — $0, built into web sprint - P2: Full PWA — service worker, offline cache, push notifications — 2 weeks - P3: PWA enhancements — background sync, IndexedDB offline store — 1 week - P4: React Native app — only if traction warrants — 8–12 weeks

    Consequences

    • Positive: Zero additional code for Phase 0–1. No app store friction. Mobile value from day 1 of web launch.
    • Negative: PWA push notifications are less reliable than native on iOS. Some enterprise deals may require native apps.
    • Mitigation: PWA foundation means we don't start from zero for React Native.

    ADR-006: Auth — Clerk

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Free SMB tier, pre-built React UI, Google/MS social login

    Context

    User registration/login, team management, RBAC, social login (Google, Microsoft — critical for email/calendar integration), MFA, future SSO (SAML/OIDC).

    Options Considered

    Provider Pricing (SMB Scale) Pre-built UI Social Login MFA Enterprise SSO
    Clerk Free tier: 10K users ✅ Excellent (React) ✅ Google, MS, GitHub, Apple ✅ ✅ (Business plan)
    Auth0 $0–$23/user/mo (B2B) ✅ Good ✅ All major ✅ ✅ Native
    Supabase Auth Free tier generous ✅ Good ✅ Google, MS, GitHub ⚠️ Limited ❌ No SAML
    Custom OAuth $0 (dev cost: 4–6 weeks) ❌ Build from scratch ❌ Must implement each ❌ Must implement ❌ Must implement

    Decision

    Clerk — selected.

    Auth architecture: - Clerk provides authn/authz via React components (SignIn, SignUp, UserProfile) - JWT tokens verified by FastAPI middleware (JWKS endpoint) - No session DB needed — Clerk manages sessions - RBAC managed in Clerk organizations → mapped to tenant_id in PostgreSQL - Google/Microsoft OAuth via Clerk social login → OAuth tokens stored in our DB for email/calendar sync - Webhook handler for user lifecycle events (user.created, user.deleted, org.membership)

    Consequences

    • Positive: Free tier handles 10K users + unlimited orgs. Pre-built React components match the design system. Social login for Gmail/Outlook OAuth flow.
    • Negative: Vendor lock-in — migrating off Clerk requires rebuilding auth UI and OAuth token storage.
    • Mitigation: JWT caching means tokens are valid for their TTL even if Clerk is down. OAuth tokens for email/calendar sync live in our DB, not Clerk.

    ADR-007: Cloud Hosting — GCP

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Vertex AI for voice pipeline, serverless ops for small team

    Context

    Managed PostgreSQL and Redis, AI/ML compute (GPU for Whisper transcription), Pub/Sub for event-driven architecture, object storage, container orchestration, global low-latency, simple cost structure.

    Options Considered

    Provider Managed PG Managed Redis AI/ML Pub/Sub Containers Cost
    GCP ✅ Cloud SQL ✅ Memorystore ✅ Vertex AI, GPUs ✅ Pub/Sub ✅ Cloud Run Medium
    AWS ✅ RDS ✅ ElastiCache ✅ SageMaker ✅ SQS/SNS ✅ ECS/EKS High
    Railway ✅ Managed ✅ Managed ❌ No GPU ❌ Limited ✅ Native Low
    Render ✅ Managed ⚠️ Upstash addon ❌ No GPU ❌ No ✅ Native Low
    Fly.io ✅ Managed ✅ Upstash ❌ No GPU ❌ No ✅ Fly Machines Low-Medium

    Decision

    GCP — selected.

    Phase 0–1 architecture: - Cloud Run — serverless containers for FastAPI (auto-scaling to zero, pay per request) - Cloud SQL (PostgreSQL 16 + TimescaleDB) - Memorystore (Redis 7) — cache + Celery broker + pub/sub - Cloud Tasks — scheduled job dispatching - Cloud Pub/Sub — Gmail push notification ingestion - Cloud Storage (GCS) — file storage until CDN needed, then migrate to R2 - Secret Manager — API keys, Clerk secret key, DB passwords - Cloud Logging + Monitoring — observability

    Est. monthly cost Phase 0–1: ~$485/mo (100 tenants, 1,000 users)

    Consequences

    • Positive: Vertex AI available if we ever need to fine-tune models or self-host Whisper. Cloud Run's serverless model fits a 3-engineer team. Pub/Sub integrates natively with Gmail push notifications.
    • Negative: GCP learning curve for the team. Cost management is not trivial — must set budget alerts from day 1. GCS egress costs higher than R2.
    • Mitigation: Cloud Run's simplicity reduces ops overhead. R2 used for user-downloaded files to avoid egress costs.

    ADR-008: Email Delivery — Resend + SES fallback

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: React Email templates, good deliverability, cost-effective bulk routing

    Context

    Multiple email types: transactional (deal notifications, task reminders), bulk (drip campaigns, onboarding sequences), system (password resets), mirrored (via Gmail/Outlook APIs, not SES).

    Options Considered

    Provider Deliverability API DX Pricing (100K/mo) Templates
    Resend ✅ Excellent ✅ Best (React Email) $20/mo (100K) ✅ React Email
    SendGrid ✅ Good ⚠️ Complex ~$20/mo (50K cap) ✅ Drag-drop
    AWS SES ⚠️ Requires warming ❌ Low-level API $10/mo (100K) ❌ Raw

    Decision

    Resend (primary) + SES (bulk fallback) — selected.

    • Transactional, notification, and invite emails go through Resend
    • SES as bulk routing fallback via Resend's built-in routing
    • React Email templates shared with the frontend codebase
    • Delivery webhooks update email engagement analytics in the CRM

    Consequences

    • Positive: Best-in-class developer experience. React Email templates are type-safe and testable. Good deliverability without reputation warming.
    • Negative: Resend is younger (founded 2022) than SendGrid/SES. Vendor lock-in for templates.
    • Mitigation: React Email is framework-agnostic — switching providers is straightforward. Start SES warmup process at Phase 1 even if unused.

    ADR-009: Search — Meilisearch

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Out-of-box typo tolerance, faceted search, minimal ops

    Context

    Search across contacts (name, email, phone, company), deals (name, value, stage, notes), emails (subject, body, sender), notes and activities, files. Requires typo tolerance, faceted filtering, fast autocomplete, relevance ranking.

    Options Considered

    Engine Setup Time Typo Tolerance Faceted Search Operations Cost
    Meilisearch 15 min ✅ Excellent (built-in) ✅ Native Minimal Free (self-host)
    Typesense 30 min ✅ Good ✅ Configurable Minimal Free (self-host)
    Elasticsearch 1–2 days ⚠️ Requires config ✅ Powerful Heavy Free (expensive infra)
    PostgreSQL FTS 0 min ❌ None ⚠️ Partial None $0

    Decision

    Meilisearch — selected.

    • Self-hosted on GCP (or Meilisearch Cloud at $99/mo for zero ops)
    • One index per entity: contacts, deals, emails, notes
    • Sync via Celery worker — on entity create/update, trigger index update
    • Faceted filters: stage, owner_id, deal_value_range, created_date
    • Search API exposed through FastAPI (auth middleware ensures tenant isolation)
    • PostgreSQL FTS as fallback if Meilisearch is down

    Consequences

    • Positive: Zero-config typo tolerance, built-in faceted search with no index mapping needed, minimal ops overhead.
    • Negative: Must stay in sync with PostgreSQL (eventual consistency acceptable — sub-second latency). Not suitable for complex aggregations — use PostgreSQL/TimescaleDB for those.

    ADR-010: File Storage — Cloudflare R2

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: S3-compatible API, zero egress fees, CDN delivery

    Context

    Email attachments, deal documents, profile images, call recordings, report exports. Files need secure access (signed URLs), preview support, CDN delivery.

    Options Considered

    Provider S3-Compatible Egress Fees CDN Signed URLs Pricing (1TB)
    Cloudflare R2 ✅ Yes $0 (no egress) ✅ Global CDN ✅ Yes ~$15/mo
    AWS S3 ✅ Native ~$0.09/GB ✅ CloudFront ✅ Yes ~$23/mo + egress
    GCS ✅ Yes ~$0.12/GB ✅ Cloud CDN ✅ Yes ~$20/mo + egress

    Decision

    Cloudflare R2 — selected.

    • S3-compatible API (boto3/S3 SDK with R2 endpoint)
    • Signed URLs for secure client access (time-limited, per-file)
    • Pre-signed upload URLs for direct client upload (or proxy through FastAPI for virus scanning)
    • Image thumbnails via Celery worker (Pillow)
    • Call recording files encrypted at rest

    Consequences

    • Positive: Zero egress fees — significant savings for a CRM where users frequently download attachments. Global CDN included automatically.
    • Negative: Fewer regions than S3 (but Cloudflare's global network compensates).
    • Neutral: Migrating to S3 later is trivial (same API). Must configure CORS policies for direct browser upload.

    ADR-011: Task Queue — Celery + Redis

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Python-native, Redis already in stack, mature ecosystem

    Context

    Extensive background processing: email sync (Gmail/Outlook polling, delta sync, history processing), calendar sync (event sync, recurrence expansion), voice processing (ingestion, transcription, analysis), webhook delivery, report generation, file processing (thumbnails, virus scan), scheduled tasks (daily digests, pipeline snapshots).

    Options Considered

    Queue Language Fit Reliability Scheduling Monitoring Setup
    Celery + Redis ✅ Python-native ✅ Good (acks, retries, DLQ) ✅ Celery Beat ✅ Flower / Prometheus Redis already in stack
    BullMQ + Redis ✅ Node-native ✅ Good ✅ Cron jobs ✅ Bull Board Python backend ❌
    Cloud Tasks (GCP) ✅ Any HTTP ✅ Excellent ❌ No cron ✅ GCP Console No Redis needed
    RabbitMQ ✅ Any AMQP ✅ Excellent ⚠️ Via separate scheduler ✅ Management UI Separate infra

    Decision

    Celery + Redis — selected.

    Architecture: - Broker: Redis (already in stack) - Result backend: Redis - Beat scheduler: Celery Beat for recurring tasks - Separate queues per workload: email_sync, calendar_sync, voice, webhooks, reports, files, default - Concurrency: Prefork workers for CPU-bound AI work, Gevent workers for I/O-bound sync work - Monitoring: Flower (web UI) + Prometheus metrics

    Consequences

    • Positive: Single language (Python) for backend + queue logic. Redis already in stack. Mature ecosystem for complex workflows (chains, groups, chords for voice processing pipeline).
    • Negative: Celery + Redis adds operational complexity vs managed queues. Celery's documentation is vast but uneven.
    • Mitigation: GCP Memorystore provides managed Redis. Start simple (single queue, default config) and add routing as task volume grows. Use CELERY_ACKS_LATE with persistent Redis to avoid task loss.

    ADR-012: Sync Engine — Unified Engine with Adapter Pattern

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Provider-agnostic core, delta-first sync, OAuth lifecycle

    Context

    Bidirectional sync between CRM data and external providers (Gmail, Microsoft Graph, Google Calendar, Outlook Calendar). Must handle email sync, calendar sync, bidirectional changes, delta sync, and conflict resolution.

    Providers: | Provider | Email API | Calendar API | Sync Method | Complexity | |----------|-----------|-------------|-------------|------------| | Gmail | Gmail API v1 + Pub/Sub | Google Calendar API v3 | historyId / syncToken | Medium-High | | Microsoft Graph | Graph /me/messages/delta | Graph /me/calendarView/delta | deltaLink | Medium | | IMAP/SMTP | IMAP UID + CONDSTORE | CalDAV | UIDVALIDITY/MODSEQ | High |

    Options Considered

    1. Build in-house — Unified sync engine with adapter pattern. Cost: 6–10 weeks. Full control. $0 per-user.
    2. Sync middleware (Nylas/Cronofy) — $2–5/user/mo. Time to market: 1–2 weeks. Data privacy concern (email bodies through third party).
    3. Hybrid (selected) — Build native adapters for Graph + Google (covering 80%+). Middleware for IMAP/CalDAV long tail.

    Decision

    Hybrid approach — selected.

    Build native adapters for Gmail API, Google Calendar API, and Microsoft Graph. Use middleware (Nylas or similar) for IMAP/CalDAV in Phase 2 when demand materializes.

    Sync engine architecture: - Async everything — all sync via Celery workers. Never block an HTTP request on third-party API calls. - Delta-first sync — full sync only on initial connection. Normal operation is incremental via historyId/deltaLink. - Eventually consistent — 30–60s sync latency is acceptable for SMB segment. - Idempotent — running the same sync twice produces the same result. - Conflict resolution — last-write-wins initially, with full conflict logging. - OAuth lifecycle — token refresh failures surfaced to users. Expired connections paused, not silently broken. - Tiered sync intervals — Free: 120s polling. Starter: 60s. Growth: 30s + webhooks. Enterprise: real-time webhooks.

    Consequences

    • Positive: Full control over conflict resolution, data model, OAuth flow. No per-user middleware fees for primary providers.
    • Negative: 6–10 weeks development for primary adapters. Maintenance burden for provider API changes.
    • Mitigation: Adapter pattern means adding a new provider is a well-defined interface implementation. Webhooks from Gmail/Graph reduce polling load.

    ADR-013: Architecture — Modular Monolith

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Team of 3–5 engineers cannot maintain 12 microservices

    Context

    A team of 3–5 founding engineers must ship a working CRM in 3 months. Microservices add deployment complexity, operational overhead, and require significant infrastructure investment.

    Options Considered

    Approach Complexity Team Fit Evolution Path Cost
    Modular monolith Low ✅ Natural for 3–5 Extract services as team grows Low
    Microservices High ❌ Requires 8+ engineers Already there High
    Serverless functions Medium ⚠️ Works for stateless Hard to extract Medium

    Decision

    Modular monolith with strict domain boundaries — selected.

    Architecture: - Single FastAPI application with clear domain modules: contacts, activity, auth, sync, voice, search, reports, webhooks, api - Each module has its own models, schemas, services, and routes - Modules communicate via function calls (within the same process), not HTTP - Extract to services only when latency or team-size demands it - Celery workers handle background processing (separate process, same codebase)

    Consequences

    • Positive: Simple deployment (single container). Fast iteration speed. No networking complexity. Shared connection pool.
    • Negative: Risk of tight coupling between modules — mitigated by strict import boundaries and code review.
    • Neutral: Extracting a module to a microservice later is straightforward since the domain boundaries already exist.

    ADR-014: Conversation Intelligence — Build on Twilio + Deepgram

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: $0.013/min vs $100K/yr Gong — biggest whitespace opportunity

    Context

    Conversation intelligence (call recording, transcription, sentiment analysis, talk ratio, keyword spotting, real-time coaching) is the single biggest whitespace opportunity for SMB/mid-market CRM. Gong ($100K+/year) and Chorus ($50K+/year) are enterprise-only. HubSpot has zero native CI.

    Options Considered

    Approach Cost Time to Market Differentiation Data Control
    Build on Twilio + Deepgram + LLM $0.013/min (Twilio) + API costs 2–4 weeks (Tier 1) ★★★★★ Breakthrough ✅ Full
    Gong/Chorus integration $50K–100K+/yr 1–2 weeks ★ Low (reselling) ❌ Third party
    Buy middleware (Aircall + basic CI) $30/user/mo 4–6 weeks ★★ Low ⚠️ Limited

    Decision

    Build on Twilio + Deepgram + LLM — selected.

    Phased implementation: - Tier 1 (MVP+): Call recording + transcription via Twilio Media Streams → Deepgram API ($0.013/min). 2–4 weeks. - Tier 2 (Year 1): Conversation intelligence — sentiment analysis, talk ratio, keyword spotting via fine-tuned BERT + LLM. 2–3 months. ★★★★★ Unique in SMB. - Tier 3 (Year 2): Real-time coaching prompts — Whisper live transcription → lagre language model-based suggestion engine. 3–6 months. - Tier 4 (Year 2+): Deal risk scoring from call analysis. 4–6 months.

    Pricing: Offer CI at $29–49/user/mo — 100x cheaper than Gong.

    Consequences

    • Positive: Full data control and privacy. Buildable with open-source components (Whisper, BERT, spaCy). Cost-effective at scale with shared model serving. LLM costs dropping to <$0.01/call.
    • Negative: Significant engineering investment (6–9 months for full stack). Real-time coaching is technically complex.
    • Mitigation: Start with Tier 1 (table stakes) and iterate. Pluggable telephony layer: Twilio v1, Aircall v2, Vonage v3.

    ADR-015: Integration — Adapter Pattern with Hybrid Sync

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Atlas
    Driver: Build native for 80% market coverage, middleware for the long tail

    Context

    28 integration providers across 11 categories. A team of 3–5 engineers must prioritize which to build natively and which to cover via third-party middleware (Zapier, Make, n8n).

    Integration Strategy

    Build native adapters for integrations covering >30% market share; use Zapier/Make for everything else; avoid building for <5% market share.

    Day 1 (P0): Gmail + Microsoft Graph (email/calendar), Meta Lead Ads, Zapier/Make connectors, Clerk Month 2–3 (P1): Stripe, Shopify, QuickBooks, Xero, Slack, Twilio, Mailchimp, GA4 Month 3–4 (P2): IMAP middleware, HubSpot Marketing, LinkedIn, Zoom, Teams Future (P3+): Aircall, RingCentral, Twitter/X, FreshBooks, WooCommerce, Mixpanel, Tableau

    Decision

    Adapter pattern + tiered build strategy — selected.

    • Each integration is a pluggable adapter with a common interface (sync(), webhookHandler(), oauthRefresh())
    • Unified sync engine schedules all active integrations per tenant
    • Normalized data model maps provider-specific schemas to FrontierCRM's domain
    • OAuth token vault with auto-refresh
    • Webhook router — single HTTPS endpoint dispatching to the correct adapter
    • Provider-aware rate limiter with automatic backoff

    Consequences

    • Positive: Adding a new integration is implementing one adapter interface. Consistent error handling, logging, and monitoring across all integrations.
    • Negative: Each native adapter requires ongoing maintenance for provider API changes. 6–10 weeks to reach Day 1 integration coverage.
    • Mitigation: Zapier covers the long tail (9,000+ apps) with zero engineering effort. Adapter pattern makes deprecation graceful.

    ADR-016: Pricing Philosophy — All-Inclusive with AI Included

    Status: ✅ Accepted
    Date: 2026-06-28
    Author: Scout
    Driver: Transparent pricing is #1 market gap; AI-as-upsell is #2 user complaint

    Context

    Every major CRM has confusing pricing tiers, feature gates, and hidden costs. Salesforce users report base price is 30% of actual total. HubSpot's multi-hub pricing forces $1,600+/mo for basic features. AI features are gated behind +$50-100/user/mo add-ons (Einstein, Copilot, Breeze, Freddy).

    Pricing Principles

    1. All features included in every paid tier — No feature gates, no hidden SKUs, no consumption-billing surprises.
    2. AI-native, not AI-bolted-on — AI included at every paid tier, not upsold. Lead scoring, email drafting, meeting summaries, predictive forecasting — all in the base price.
    3. Transparent, predictable pricing — Single per-user price. No consumption credits, no data cloud metering, no surprise overages.
    4. Unified platform — one product — Don't unbundle into Sales/Marketing/Service hubs. One product, one price, all modules included.
    5. Setup in 10 minutes — The CRM should be usable on day 1 with minimal configuration. No consultant needed.

    Decision

    5-tier all-inclusive model — selected.

    Tier Price Target Key Differentiator
    Free $0 (2 users) Micro business, evaluators No time limit, no feature crippling
    Starter $19/user/mo 2–25 employees, SMB AI included (competitors gate it)
    Growth $49/user/mo 10–100 employees, mid-market AI meeting summarization + workflow builder
    Pro $99/user/mo 50–500 employees Conversation intelligence + CPQ + SSO
    Enterprise Custom 500+ employees White-label, dedicated infra, data residency

    Annual discount: 18%. 3-year ARR projection: $0.9M → $8.2M → $34.5M.

    Consequences

    • Positive: Transparent pricing is FrontierCRM's strongest GTM differentiator. AI included at every paid tier eliminates the #2 user complaint across all competitors. Starter tier at $19/user/mo undercuts HubSpot ($20 + separate hubs) and undercuts Salesforce Starter ($25 + no AI).
    • Negative: AI inference costs are real — mitigated by AI Credits system for usage-based metering at scale. Gross margin pressure from including AI in base price (72% Y1 → 82% Y3).
    • Risk: Competitors may drop prices — but their enterprise cost structures make this hard. FrontierCRM's lean stack (~$485/mo infra) enables margin at lower prices.
