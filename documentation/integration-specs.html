    CRM Communication & Telephony Integration Specifications

    Target Audience: SMB CRM (10-200 employees) Last Updated: June 2026

    Executive Summary

    This document provides structured integration specifications for six communication and telephony platforms critical to a modern CRM. Each section covers: integration pattern, authentication, capabilities, rate limits, sync complexity, common pain points, and strategic recommendations for an SMB-focused CRM (10-200 employees).

    1. Slack API

    1.1 Integration Pattern

    Property Detail
    Primary Interface REST API (api.slack.com)
    Real-time Events API (webhooks) or Socket Mode (WebSocket)
    SDKs Bolt (JS/Python/Java), Slack SDK (Python/JS/Java/Go/.NET)
    Legacy RTM API (deprecated), Incoming Webhooks (stable)

    1.2 Authentication and OAuth 2.0

    Token Type Description Use Case
    Bot Token (xoxb-) Represents the bot user, scoped per workspace Sending messages, reading channels
    User Token (xoxp-) Represents an installing user User-level actions (profile, DMs)
    App Token (xapp-) Used for Socket Mode auth Socket Mode connections
    Incoming Webhook URL Pre-configured URL for posting to single channel Simple notifications

    OAuth Flow: 1. User clicks "Add to Slack" -> redirected to Slack OAuth consent page 2. User approves requested scopes (e.g., channels:read, chat:write, users:read) 3. Slack redirects back with a temporary code 4. Server exchanges code for an access token at oauth.v2.access 5. Token stored per workspace; used for all subsequent API calls

    Key Scopes for CRM: - channels:read / channels:history -- List channels, read message history - chat:write / chat:write.customize -- Send messages with custom sender name/icon - users:read / users:read.email -- Read user profiles and emails - reactions:read -- Read message reactions - files:read -- Access shared files - links:read / links:write -- Link preview management - im:read / im:history -- Direct messages

    1.3 REST API Capabilities

    Category Key Endpoints Notes
    Channels conversations.list, conversations.info, conversations.create, conversations.invite Use Conversations API (not legacy channels.*)
    Messages chat.postMessage, chat.update, chat.delete, chat.scheduleMessage Block Kit for rich formatting
    Threads chat.postMessage with thread_ts Replies linked via thread timestamp
    Files files.upload, files.info, files.list Upload up to 1GB
    Users users.list, users.info, users.lookupByEmail Paginated; use cursor-based
    Search search.messages, search.files Requires search:read scope

    1.4 Message Formatting

    Format Description
    Plain text Simple text field
    mrkdwn Lightweight markup: bold, italic, ~strikethrough~, code,
    Block Kit JSON-based structured blocks: section, header, divider, image, actions, input, context
    Block Kit Builder Visual tool at app.slack.com/block-kit-builder

    1.5 Events API (Real-time Webhooks)

    • Subscribe to events like message.channels, message.im, channel_created, member_joined_channel
    • Events are POSTed to a public HTTPS endpoint
    • Must respond with HTTP 200 within 3 seconds (otherwise retried)
    • URL verification challenge on setup
    • Rate limit: 30,000 event deliveries per workspace/app per 60 minutes
    • Retry: Exponential backoff (3 attempts), then dropped

    1.6 Socket Mode

    • Alternative to public webhook endpoint -- establish persistent WebSocket connection
    • No need for public-facing HTTPS URL
    • Required for apps in development or behind firewalls
    • Uses App Token (xapp-) with connections:write scope
    • Subscribe to same events as Events API

    1.7 Rate Limits (Per Workspace Tiering)

    Tier Limit Example Methods
    Tier 1 ~1-5 per minute admin.*, users.profile.set
    Tier 2 20+ per minute conversations.info, users.info
    Tier 3 50+ per minute conversations.list, users.list (with pagination)
    Tier 4 100+ per minute Bulk operations
    Special Varies chat.postMessage (~1/s per channel + workspace-wide)
    Events API 30,000 per workspace/app/60min Event deliveries

    1.8 Slack Connect Considerations

    • Shared Channels that span multiple workspaces/organizations
    • Channel ID changes: When a private channel becomes shared, ID prefix may change from G to C
    • External members: User objects from external teams have is_stranger: true flag and limited profile data
    • Bot DMs: A bot can DM users on all connected workspaces if in a shared channel together
    • Events: Subscribe to channel_shared / channel_unshared to track sharing state
    • Conversations API: Must use conversations.* methods -- legacy APIs don't work with shared channels

    1.9 Common Pain Points

    Pain Point Impact Mitigation
    Rate limits hit during bulk operations Failed messages, 429 errors Implement exponential backoff; batch writes across channels
    Token management per workspace State complexity Use a token store (DB/kv) keyed by team_id
    Event duplicate delivery Duplicate CRM activity logs Idempotency keys / dedup on event_ts
    Slack Connect external user data Incomplete profiles Handle is_stranger gracefully; don't require locale
    Block Kit complexity Steep learning curve Use templates; build a UI component for Block Kit composition

    1.10 Strategic Recommendations (SMB CRM)

    • Priority: High. Slack is the #1 communication tool for SMBs. Deep integration is table stakes.
    • Start with: Bot token + Events API. Send deal updates, activity notifications, and @mentions from CRM.
    • Message patterns: In-channel notifications for team deals; DM notifications for personal tasks/reminders.
    • Avoid: Full message sync (fetching all history) -- use search API sparingly.
    • Recommended SDK: Bolt (JS/Python) -- handles event routing, acknowledgments, and retries.
    • Testing: Use Slack's app-manifest.yaml for repeatable app setup across dev/staging/prod.

    2. Microsoft Teams

    2.1 Integration Pattern

    Property Detail
    Primary Interface Microsoft Graph API (graph.microsoft.com)
    Real-time Graph change notifications (webhooks) + Bot Framework
    SDKs Microsoft Graph SDK (.NET/JS/Python/Java/Go), Bot Framework SDK
    Auth OAuth 2.0 via Azure AD (Microsoft Entra ID)

    2.2 OAuth Scopes (Microsoft Graph)

    Scope Permission Description
    Channel.ReadBasic.All Delegated Read channel names and descriptions
    ChannelMessage.Read.All Application Read all channel messages
    ChannelMessage.Send Application Send channel messages
    Chat.ReadWrite.All Application Read/write 1:1 and group chats
    TeamsTab.ReadWriteAll Application Manage tabs in channels
    OnlineMeetings.ReadWrite.All Application Create/read meetings
    User.Read.All Delegated Read user profiles
    Presence.Read.All Application Read user presence

    2.3 Graph API Capabilities

    Category Endpoint Notes
    Teams /teams/{id} List, get, create teams
    Channels /teams/{id}/channels CRUD channels; standard vs private channels
    Messages /teams/{id}/channels/{id}/messages Send/reply; supports rich HTML content
    Chats /chats/{id}/messages 1:1 and group chat messages
    Tabs /teams/{id}/channels/{id}/tabs Add tabs (apps/web content) to channels
    Meetings /users/{id}/onlineMeetings Create/update/list online meetings
    Calendar /users/{id}/events Calendar integration
    Presence /users/{id}/presence Real-time availability

    2.4 Bot Framework Integration

    • Microsoft Bot Framework SDK -- Build conversational bots that work in Teams, Slack, Web, etc.
    • Teams Activity Handler -- Extends Bot Framework for Teams-specific features
    • App Manifest (manifest.json): Defines bot identity, scopes (personal/team/groupChat), commands, static tabs
    • Deployment: Bots must be hosted on a public HTTPS endpoint; use Bot Framework's connector service

    2.5 Webhooks for Notifications

    Type Description
    Change Notifications (webhooks) Subscribe to resource changes (messages, members) via /subscriptions
    Outgoing Webhooks Simple POST-to-URL when @mentioned in a channel
    Connectors Legacy; POST to Teams channel via webhook URL

    2.6 Rate Limits

    Limit Scope Value
    Requests per 10 seconds Per app per tenant Varies by API (typically 1300-4000)
    Requests per 10 seconds Per app per resource 300-1000
    Read throttling messages endpoint 5 requests per second per channel
    Subscription renewal subscriptions Max 3 day lifetime; must renew

    2.7 Challenges (Tenant Admin Consent)

    Challenge Impact Mitigation
    Admin consent required App fails if tenant admin hasn't pre-approved Implement admin consent URL
    Permission delegation Users can't self-approve app-level permissions Design around delegated permissions where possible
    Multi-tenant considerations App must be multi-tenant registered for ISV Register in Azure AD as multi-tenant app
    Conditional Access policies Enterprise customers may block scopes Handle auth failures gracefully
    Graph API changes Breaking changes on beta endpoints Use v1.0 for production

    2.8 Common Pain Points

    Pain Point Impact Mitigation
    Admin consent friction High drop-off during onboarding Provide clear admin consent step-by-step guide
    Complex auth flow Developer confusion Use MSAL libraries; implement token caching
    Private channel restrictions App can't read private channels without explicit membership Note in docs that private channels are limited
    Change notification 3-day expiry Requires cron job to renew subscriptions Implement auto-renewal loop
    Rate limits on message reads Slow channel history sync Batch reads into smaller time windows
    Bot Framework complexity Steeper learning curve Start with outgoing webhooks for simple CRM alerts

    2.9 Strategic Recommendations (SMB CRM)

    • Priority: Medium-High. Teams adoption is growing in SMB, but admin consent is a barrier for self-serve onboarding.
    • Start with: Outgoing webhooks / Office 365 Connectors for CRM notifications (deal updates, task assignments). These require no admin consent.
    • Later: Bot Framework for interactive features (slash commands, message actions). This requires admin consent.
    • Avoid: Full bidirectional message sync -- overly complex. Focus on CRM -> Teams notifications first.
    • Auth approach: Offer both delegated (user signs in) AND app-only (admin pre-consents) registration paths.
    • Pricing: Requires Microsoft 365 Business Basic/Standard/Premium -- your SMB customers likely have this.
    • Graph SDK: Use Microsoft Graph SDK for JS/Python -- handles auth, pagination, retries.

    3. Zoom API

    3.1 Integration Pattern

    Property Detail
    Primary Interface Zoom REST API (api.zoom.us/v2)
    Real-time Zoom Webhooks (Event Subscriptions)
    SDKs Zoom SDK (JS/Android/iOS), Zoom Meeting SDK, Zoom Video SDK
    Auth OAuth 2.0 (Server-to-Server OAuth or Account-Level), JWT (legacy)

    3.2 Authentication

    Auth Type Tokens Use Case
    OAuth 2.0 (Server-to-Server) Access token (valid 1 hour) Server-side app; no user interaction
    OAuth 2.0 (Account-Level) Access + Refresh tokens App installed by Zoom account admin
    OAuth 2.0 (User-Level) Access + Refresh tokens App installed by individual Zoom user
    JWT (Legacy) Self-generated JWT Deprecated; avoid new builds

    3.3 REST API Capabilities

    Category Endpoints Notes
    Users GET /users, GET /users/{id}, PATCH /users/{id} List, get, update users
    Meetings POST /users/{id}/meetings, GET /meetings/{id}, PATCH /meetings/{id}, DELETE /meetings/{id} Full CRUD; supports recurring meetings
    Meeting Registrants GET /meetings/{id}/registrants, POST /meetings/{id}/registrants Registration for webinars/meetings
    Webinars POST /users/{id}/webinars, GET /webinars/{id} Requires webinar license
    Recordings GET /users/{id}/recordings, GET /meetings/{id}/recordings, DELETE /meetings/{id}/recordings List, download, delete cloud recordings
    Reports GET /report/meetings/{id}, GET /report/users/{id}/meetings Meeting reports, participant details

    3.4 Webhooks (Real-time Events)

    Event Category Events Notes
    Meeting meeting.started, meeting.ended, meeting.registration_created Core lifecycle events
    Participant meeting.participant_joined, meeting.participant_left Track attendance
    Recording recording.completed, recording.transcript_completed Recording ready notification
    Webinar webinar.started, webinar.ended, webinar.registration_created Webinar lifecycle
    User user.created, user.updated, user.activated User lifecycle

    3.5 Rate Limits

    Plan / API Limit Notes
    Free/Basic 1 req/sec per dashboard Heavily restricted
    Pro/Business/Enterprise ~10-30 req/sec per dashboard Varies by account tier
    OAuth token generation 10 requests per minute Across all apps for an account
    Concurrent webhook deliveries ~50/second Queued if exceeded

    3.6 Common Pain Points

    Pain Point Impact Mitigation
    Zoom license requirements Can't create meetings for users without paid license Check license status; offer fallback (send join link only)
    OAuth token expiry (1 hour) Stale tokens cause API failures Implement robust token refresh with retry
    Rate limits on free tier Unusable for CRM at scale Recommend customers have Pro+ licenses
    Webhook verification Security risk Always verify Zoom-Verification-Token header
    Timezone handling Meeting times display incorrectly Store UTC + IANA timezone
    Recording download auth Recordings need auth to download Use OAuth token in URL; manage download queue

    3.7 Strategic Recommendations (SMB CRM)

    • Priority: Medium. Zoom is widely used, but most CRM meeting needs can be met by link-sharing without full API integration.
    • Start with: Read-only integration: list upcoming meetings, show join_url in CRM records.
    • Auto-schedule meetings: CRUD meetings from CRM. Store Zoom meeting ID + join URL on contact/lead/deal records.
    • Attendance tracking: Use meeting.participant_joined and meeting.ended webhooks to log call activity in CRM.
    • Avoid: Real-time video streaming inside CRM. Leave that to the Zoom client.
    • OAuth choice: Use Server-to-Server OAuth (simplest for CRM backend).
    • Licensing check: Always verify the Zoom user has a Pro+ license before creating meetings via API.

    4. RingCentral API

    4.1 Integration Pattern

    Property Detail
    Primary Interface RingCentral REST API (platform.ringcentral.com)
    Real-time PubNub-based event notifications + Webhooks
    SDKs RingCentral SDK (.NET/JS/PHP/Python/Ruby/Java)
    Auth OAuth 2.0 (Authorization Code + JWT)

    4.2 Authentication

    Auth Flow Description Use Case
    Authorization Code 3-legged OAuth. User logs in, approves scopes, gets code -> token App installed per-user
    JWT Bearer Server-to-server with a JWT issued by RingCentral Backend integration; no user interaction
    Legacy (password) Resource Owner Password Grant Deprecated

    4.3 REST API Capabilities

    Category Key Endpoints Notes
    Call Log GET /account/{id}/call-log, GET /account/{id}/extension/{id}/call-log Paginated; filter by date, type (voice/sms/fax)
    SMS POST /account/{id}/extension/{id}/sms Send SMS; requires outbound SMS permissions
    Presence GET /account/{id}/extension/{id}/presence User presence (Available/Busy/DND/Offline)
    Call Recordings GET /account/{id}/recording/{id} Download call recordings
    Voicemail GET /account/{id}/extension/{id}/voicemail List/retrieve voicemail messages
    Contacts GET /account/{id}/extension/{id}/address-book/contact, POST/PUT/DELETE Sync contacts
    Extensions (Users) GET /account/{id}/extension, GET /account/{id}/extension/{id} List users/extensions
    Call Queues GET /account/{id}/call-queues Manage call queues
    Call Forwarding PUT /account/{id}/extension/{id}/forwarding-number Manage call forwarding rules

    4.4 Webhooks (Event Notifications)

    RingCentral uses two delivery mechanisms:

    Method Description
    PubNub (legacy) Real-time via PubNub channels; one channel per extension
    Webhooks (recommended) POST to HTTPS endpoint; set up via Subscription API

    4.5 Rate Limits

    Limit Scope Value
    API requests Per app per account 40 requests per second (standard); 150+ (enterprise)
    SMS sending Per extension 30 SMS per minute per user
    Subscriptions Per app per extension Max 20 active subscriptions
    Subscription duration Per subscription Max 23 hours (must renew)

    4.6 Common Pain Points

    Pain Point Impact Mitigation
    Subscription renewal frequency (23hr) Requires cron job Implement auto-renewal loop in background worker
    Phone system complexity Many config options overwhelm devs Abstract with simplified CRM models
    Legacy PubNub vs. Webhook confusion Wrong architecture choice Use webhooks; avoid PubNub for new builds
    Call recording download auth Recordings need auth token in URL Implement secure download proxy
    Presence data granularity DND vs. OnCall confusion Map to simpler CRM availability states
    Extension vs. User model Different structures per account Normalize to CRM's user model

    4.7 Strategic Recommendations (SMB CRM)

    • Priority: Medium. RingCentral is a solid VoIP provider but less common than Twilio in SMB (more common in mid-market).
    • Start with: Call log sync (inbound/outbound calls to CRM activity records), SMS sending from CRM, and click-to-dial.
    • Webhook-first: Use webhooks for real-time SMS/call alerts in CRM.
    • Avoid: Full presence webhook streaming (too noisy). Poll presence only when needed.
    • JWT auth: Prefer JWT bearer over Authorization Code -- simpler for backend CRM integration.
    • Contact sync: Offer one-way (RingCentral -> CRM) with address book API.
    • Pricing: Requires RingCentral Office/Pro licenses. Worth offering as an add-on, not core.

    5. Twilio

    5.1 Integration Pattern

    Property Detail
    Primary Interface Twilio REST API (api.twilio.com)
    Real-time Webhooks (Status Callbacks, Incoming Webhooks)
    SDKs Twilio Helper Libraries (Python/JS/Java/C#/PHP/Ruby/Go/Node)
    Auth HTTP Basic Auth (Account SID + Auth Token) + API Keys

    5.2 Authentication

    Method Details
    Account SID + Auth Token Primary auth. Sent as HTTP Basic Auth on every request.
    API Keys Create via Console. Fine-grained permissions (Main/Restricted).
    API Key + Secret Recommended for production -- easier to rotate and scope per integration.

    5.3 Phone Number Management

    Operation Endpoint Details
    Search available numbers GET /2010-04-01/Accounts/{sid}/AvailablePhoneNumbers/{country}/Local.json Filter by area code, capabilities
    Purchase number POST /2010-04-01/Accounts/{sid}/IncomingPhoneNumbers.json Auto-assign available number
    List owned numbers GET /2010-04-01/Accounts/{sid}/IncomingPhoneNumbers.json Paginated; filter by phone number
    Release number DELETE /2010-04-01/Accounts/{sid}/IncomingPhoneNumbers/{sid}.json Releases number back to pool
    Configure number POST /2010-04-01/Accounts/{sid}/IncomingPhoneNumbers/{sid}.json Set voice/SMS URLs, friendly name

    5.4 SMS / Messaging

    Operation Endpoint Details
    Send SMS POST /2010-04-01/Accounts/{sid}/Messages.json From (Twilio number), To, Body; supports MediaUrl for MMS
    Send to multiple POST /2010-04-01/Accounts/{sid}/Messages.json (per recipient) Notify service for batch messaging
    List messages GET /2010-04-01/Accounts/{sid}/Messages.json Date-sorted; paginated
    Get message GET /2010-04-01/Accounts/{sid}/Messages/{sid}.json Status, direction, price, error code
    WhatsApp Same endpoint, From with whatsapp:+number Requires WhatsApp Business Account
    Conversations API v1/Conversations, v1/Conversations/Messages Multi-channel conversations model

    5.5 Voice Calls

    Operation Endpoint Details
    Make call POST /2010-04-01/Accounts/{sid}/Calls.json From, To, Url (TwiML endpoint) or Twiml (inline)
    List calls GET /2010-04-01/Accounts/{sid}/Calls.json Filter by status, date, direction
    Call recordings GET /2010-04-01/Accounts/{sid}/Recordings.json WAV/MP3 files; linked to calls via CallSid
    Call transcriptions POST /2010-04-01/Accounts/{sid}/Recordings/{sid}/Transcriptions.json Async transcription; callback on completion
    Conference calls POST /2010-04-01/Accounts/{sid}/Conferences.json Conference TwiML verb

    5.6 TwiML (Twilio Markup Language)

    • XML-based instructions for Twilio's servers
    • Key verbs: Say (text-to-speech), Play (audio file), Gather (collect DTMF), Record (record call), Dial (connect to number), Conference (multi-party), Redirect, Hangup, Message (SMS during voice call)
    • TwiML Bins: Host simple TwiML without a server (for prototyping)

    5.7 Real-time Events (Webhooks / Status Callbacks)

    Event Type Triggered When
    Incoming voice Twilio receives incoming call to your number
    Incoming SMS Twilio receives incoming text
    Call status queued, ringing, in-progress, completed, busy, failed, no-answer, canceled
    Recording Recording completed + available for download
    Transcription Transcription completed
    Messaging SMS delivered, failed, read (read receipts)

    5.8 Rate Limits

    Product Limit Details
    API Requests 60 requests per second per account (default) Can request increase via support
    SMS Messaging 1 message per second per number Segment for high volume; use messaging service
    Voice Calls 1 call per second per number Queue calls if needed
    Phone Number Search 20 requests per minute Avoid polling aggressively

    5.9 Pricing Model

    Service Pricing (Approx) Notes
    Phone Numbers $1-$15/month per number Depends on country/type
    SMS (US) $0.0079 per message outbound Inbound is free; varies by carrier
    Voice (US) $0.013 per minute inbound; $0.014 outbound Per-minute billing
    Recordings $0.002 per minute (storage) + $0.01/min (transcription)
    WhatsApp $0.005-$0.08 per conversation Conversation-based (24-hour session)
    Free Trial $15 credit Enough to build + test integration

    5.10 Common Pain Points

    Pain Point Impact Mitigation
    Per-number rate limits High-volume customers hit limits Use Messaging Service with multiple sender pools
    TwiML latency Voice quality suffers if response is slow Host TwiML endpoint close to Twilio's AWS regions
    Carrier filtering SMS to VoIP/landline fails Handle undelivered status; provide error messaging
    Regulatory (10DLC, A2P) SMS registration required for US A2P 10DLC Help customers register brand and campaign
    Price unpredictability Per-message/minute billing scares SMB Offer usage tracking in CRM dashboard
    Webhook signature validation Security vulnerability Make validation mandatory
    International numbers Different compliance per country Use Twilio's Compliance API

    5.11 Strategic Recommendations (SMB CRM)

    • Priority: Highest. Twilio is the de facto telephony platform for modern CRMs.
    • Core features: Click-to-dial, SMS from CRM, call logging with recordings, phone number provisioning, IVR routing
    • Pricing strategy: Pass through Twilio costs with small markup, or bundle into CRM subscription tier
    • Simplification: Provide pre-built TwiML endpoints hosted by your CRM backend
    • Compliance: Build 10DLC registration into the onboarding flow for US customers
    • SDK choice: Use Twilio Python or Node SDK in backend; Twilio Client JS SDK for browser click-to-dial
    • Avoid: Building your own SIP infrastructure. Twilio's Elastic SIP Trunking is the right abstraction

    6. Aircall

    6.1 Integration Pattern

    Property Detail
    Primary Interface Aircall REST API (api.aircall.io/v1)
    Real-time Webhooks (Event-driven notifications)
    SDKs Community-driven (no official SDK; use HTTP request libraries)
    Auth API Key (Basic Auth with api_token as username)

    6.2 Authentication

    • Every request: Authorization: Basic base64(api_id:api_token)
    • No OAuth, no token refresh. Static credentials.
    • Security note: Aircall uses a single static token per account. Token rotation requires manual dashboard update.

    6.3 REST API Capabilities

    Category Endpoints Notes
    Calls GET /calls, GET /calls/{id}, POST /calls List calls with filters, get details, create records
    Comments POST /calls/{id}/comments Add internal notes to calls
    Tags POST /calls/{id}/tags, DELETE /calls/{id}/tags/{tag_id} Tag calls for categorization
    Contacts GET /contacts, POST/PUT/DELETE Sync CRM contacts with Aircall directory
    Numbers GET /numbers, PUT /numbers/{id} List/update phone numbers
    Users GET /users, GET /users/{id} List/update Aircall users (agents)
    Teams GET /teams List Aircall teams
    Call Recordings GET /calls/{id}/recording Download call recording
    IVR GET /ivrs, PUT /ivrs/{id} Manage IVR menus (auto-attendant)
    Analytics GET /analytics/groups, GET /analytics/users Call volume, duration stats

    6.4 Webhooks

    Event Triggered When
    call.created New inbound/outbound call starts
    call.ringing_on_agent Call assigned to agent
    call.answered Agent answers the call
    call.ended Call ends (includes duration, recording URL, tags, comments)
    call.comment.created Comment added to call
    call.tag.created Tag added to call
    contact.created New contact added to Aircall
    contact.updated Contact updated in Aircall
    user.created New user added to Aircall
    user.updated User status/availability changes

    6.5 Rate Limits

    Limit Value Notes
    API requests 10 requests per second per account Hard limit across all endpoints
    Burst 30 requests in 3 seconds Exceeding triggers 429
    Pagination Max 50 per page (cursor-based) Use next_page_link

    6.6 Common Pain Points

    Pain Point Impact Mitigation
    No official SDK More boilerplate code Use generic HTTP client; create internal SDK wrapper
    Static API token Cannot rotate without downtime Implement token rotation workflow
    No webhook signature Can't verify payload origin Implement IP whitelisting + shared secret
    Low rate limit (10 req/s) Blocks bulk operations Batch reads; implement request queuing
    Limited analytics API Can't build advanced dashboards Pull raw call data; aggregate in CRM
    Contact sync complexity Two-way sync challenges Use webhook for Aircall->CRM; API for CRM->Aircall
    No IVR creation via API IVR configured in dashboard only Document as limitation

    6.7 Strategic Recommendations (SMB CRM)

    • Priority: Medium. Aircall is purpose-built for sales/customer-facing teams.
    • Core features: Click-to-dial, call activity sync via call.ended webhook, contact sync, recording/transcription storage, screen pop/CTI
    • Why choose Aircall over Twilio: Aircall is a managed phone system, not an API. For turnkey cloud phone with CRM integration, Aircall wins. For custom telephony features, Twilio wins.
    • Build a dedicated Aircall service layer in backend with caching and rate-limit management.
    • Offer Aircall as a premium integration (requires Aircall subscription). Make Twilio the default embedded telephony option.

    SMB CRM Integration Priority Ranking

    Rank Platform Rationale
    1 (Must-have) Slack SMBs live in Slack. CRM notifications + collaboration is table stakes.
    1 (Must-have) Twilio Provide embedded telephony (SMS + Voice) without forcing separate phone system vendor.
    3 (High) Zoom Meeting scheduling and attendance tracking. Most relevant for B2B sales CRMs.
    4 (Medium) Microsoft Teams Important for Microsoft-first SMBs. Start with webhooks; defer bot framework.
    4 (Medium) Aircall Good for phone-centric sales teams. Offer as premium add-on.
    4 (Medium) RingCentral Overlaps with Twilio. Focus on Twilio first. Add RingCentral per customer request.

    Recommended Architecture Approach

    +--------------------------------------------------------------+
|                     Your CRM Backend                          |
|                                                              |
|  +---------------+  +---------------+  +--------------------+  |
|  | Integration    |  | Webhook       |  | Telephony          |  |
|  | Service        |  | Handler       |  | Service (Twilio)   |  |
|  | (Slack, Zoom)  |  | (Teams, Airc) |  | (SMS, Voice, Num)  |  |
|  +-------+--------+  +-------+-------+  +----------+---------+  |
|          |                   |                      |          |
|          +-------------------+----------------------+          |
|                              |                                |
|                    +---------+---------+                     |
|                    |  Event Bus / Queue |                     |
|                    +---------+---------+                     |
|                              |                                |
|                    +---------+---------+                     |
|                    |  CRM Activity Log  |                     |
|                    +-------------------+                     |
+--------------------------------------------------------------+

    Key Design Principles: 1. Async-first: All external API calls go through a queue. 2. Webhook -> Event -> Activity: Normalized internal event -> CRM activity record. 3. Rate-limit aware: Each integration service has a retry/backoff layer. 4. Multi-tenant isolation: Tokens stored per-account, never shared. 5. Graceful degradation: Never break CRM core if external service is down.
