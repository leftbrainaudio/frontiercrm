    1. API Conventions

    Convention Rule
    Base path /api/v1/
    Protocol HTTPS only
    Content-Type application/json (request), application/json (response)
    Accept application/json
    Date format ISO 8601 with timezone: 2026-06-28T14:30:00Z
    IDs UUID v4 (string format)
    Case snake_case for properties, kebab-case for URL paths
    Versioning URL prefix (/api/v1/). Breaking changes increment version.
    Bulk operations Not yet — use individual mutations. Future: /api/v1/batch

    Standard Response Envelope

    {
  "data": { ... },            // Single resource or array of resources
  "meta": {
    "page": { ... },          // Pagination info (for list endpoints)
    "request_id": "req_abc123" // Traceable request ID
  },
  "included": [ ... ]         // Related resources (when ?include= is used)
}

    2. Authentication & Headers

    Required Headers

    Header Required Description
    Authorization: Bearer <clerk_jwt> Yes Clerk-issued JWT (verified against Clerk JWKS endpoint)
    Content-Type: application/json For mutations Request body format
    Idempotency-Key For mutations UUID — prevents duplicate processing (see Section 9)
    Accept-Language No Locale preference (default: en-US)
    User-Agent Yes Client identifier for API analytics

    Auth Flow

    Client → Clerk sign-in → Clerk JWT issued → JWT sent in Authorization header
  → FastAPI middleware verifies JWT (Clerk JWKS) →
    Extract org_id → SET app.current_tenant_id →
    Extract user_id → SET app.current_user_id →
    RBAC check → Route to handler

    RBAC Check

    The middleware extracts the user role from the JWT (Clerk org metadata) and checks against the permission matrix. Endpoints document the minimum required permission.

    Permission levels per resource: - admin — full CRUD + manage (tenant-wide) - write — create + update + delete (own or team-scoped) - read — view only - none — no access

    3. Standard Query Parameters

    These parameters work on every list endpoint unless noted otherwise.

    Parameter Type Description Default
    page[size] integer Items per page 25 (max: 100)
    page[cursor] string Cursor for cursor-based pagination —
    page[number] integer Page number (offset-based, admin only) 1
    sort string Sort field(s), - prefix = descending -created_at
    filter[field] varies Filter by field value —
    include string Comma-separated related resources —
    fields[entity] string Comma-separated field subset —
    q string Quick search (text search across key fields) —

    4. Pagination

    Default: Cursor-Based (Recommended)

    Cursor-based pagination is used for all list endpoints. It is stable (no duplicates if rows are inserted) and efficient (no OFFSET).

    Request: 

    GET /api/v1/contacts?page[size]=50&page[cursor]=eyJpZCI6IjEyMyJ9

    Response (paginated list): 

    {
  "data": [ ... ],
  "meta": {
    "page": {
      "size": 50,
      "next_cursor": "eyJpZCI6IjEyMyJ9",
      "has_more": true
    }
  }
}

    How it works: - next_cursor is a base64-encoded JSON object containing the sort key of the last item. - Client passes page[cursor] from the previous response's next_cursor. - When has_more is false, the client has reached the end. - Sort order must be stable. Default sort includes id as tiebreaker.

    Alternative: Offset-Based (Admin Endpoints Only)

    For admin panels where jumping to specific pages is needed:

    GET /api/v1/tenants?page[number]=3&page[size]=25

    Response: 

    {
  "data": [ ... ],
  "meta": {
    "page": {
      "number": 3,
      "size": 25,
      "total_pages": 42,
      "total_count": 1042,
      "has_more": true
    }
  }
}

    5. Filtering & Sorting

    Filter Syntax

    GET /api/v1/deals?filter[status]=open&filter[owner_id]=uuid&filter[value][gte]=10000
    Operator Syntax Description
    Equals filter[field]=value Exact match
    Greater than filter[field][gt]=value Numerical/date >
    Greater or equal filter[field][gte]=value >=
    Less than filter[field][lt]=value <
    Less or equal filter[field][lte]=value <=
    In filter[field][in]=a,b,c In list
    Not in filter[field][not_in]=a,b Not in list
    Contains filter[field][contains]=text Substring (text fields)
    Is null filter[field][is_null]=true NULL check
    Date range filter[created_at][gte]=2026-01-01&filter[created_at][lte]=2026-06-30 Range
    Custom field filter[custom_fields][field_key]=value JSONB custom field filter

    Sort Syntax

    GET /api/v1/deals?sort=value,-created_at
    • Ascending: sort=field_name
    • Descending: sort=-field_name
    • Multiple fields: comma-separated
    • Default: sort=-created_at (most recent first)

    Supported sort fields per endpoint are documented in the endpoint tables below.

    6. Sparse Fieldsets & Includes

    Sparse Fieldsets

    Client requests only the fields it needs:

    GET /api/v1/contacts?fields[contact]=id,first_name,last_name,email,company

    Returns only the requested fields:

    {
  "data": {
    "id": "uuid",
    "first_name": "Jane",
    "last_name": "Doe",
    "email": "...",
    "company": "Acme Corp"
  }
}

    Compound Documents (Includes)

    Client requests related resources in the same response:

    GET /api/v1/contacts?include=deals,account
GET /api/v1/deals?include=contact,stage,pipeline,activities

    Response includes an included array with the related resources:

    {
  "data": {
    "id": "uuid",
    "name": "Big Deal",
    "contact_id": "uuid",
    "pipeline_id": "uuid"
  },
  "included": [
    { "id": "uuid", "type": "contact", "first_name": "Jane", ... },
    { "id": "uuid", "type": "pipeline", "name": "Sales Pipeline", ... }
  ]
}

    Maximum include depth: 2 levels (e.g., ?include=contact.deals is not supported for performance).

    7. Error Responses

    Standard Error Envelope

    {
  "error": {
    "code": "validation_error",
    "message": "Validation failed",
    "details": [
      {
        "field": "email",
        "code": "invalid_format",
        "message": "Must be a valid email address"
      }
    ],
    "request_id": "req_abc123"
  }
}

    HTTP Status Code Map

    Code Meaning When
    200 Success GET, PATCH, DELETE
    201 Created POST (resource created)
    202 Accepted Async operation accepted (sync job started)
    204 No Content DELETE (success, no body)
    400 Bad Request Invalid JSON, missing required field, validation error
    401 Unauthorized Missing/invalid JWT
    403 Forbidden JWT valid but insufficient permissions (RBAC)
    404 Not Found Resource doesn't exist (or tenant-isolated)
    409 Conflict Idempotency key replayed, version conflict, unique constraint
    410 Gone Previous resource version deleted (delta token expired)
    422 Unprocessable Business logic error (e.g., can't move deal to won without contact)
    429 Too Many Requests Rate limit exceeded (see Retry-After header)
    500 Internal Error Unexpected server error
    502 Bad Gateway Upstream provider error (Gmail API down, Resend down)
    503 Service Unavailable Server under maintenance or overload

    8. Rate Limiting

    Strategy: Redis Sliding Window

    • Key: rate_limit:{tenant_id}:{endpoint_group}:{window}
    • Window: 60 seconds
    • Refill: per-window reset
    • Enforcement: Redis INCR + EXPIRE atomic operation

    Headers

    Every response includes rate limit headers:

    X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1687996800
Retry-After: 12          ← Only on 429 responses

    Tiers

    Tier Requests/minute Burst Applied To
    Free 60 10 Unauthenticated, free plan tenants
    Starter 300 50 Starter plan tenants
    Growth 1,000 200 Growth plan tenants
    Enterprise 5,000 1,000 Enterprise plan tenants
    Webhook 5,000 — Webhook delivery (internal)
    Admin 10,000 — Internal admin endpoints

    Rate limits are per-tenant, not per-user. A single tenant hitting the limit affects all their users.

    429 Response

    Retry-After: 12
Content-Type: application/json

{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "API rate limit exceeded. Retry after 12 seconds.",
    "retry_after_seconds": 12
  }
}

    9. Idempotency

    All mutation endpoints (POST, PATCH, DELETE) support idempotency via the Idempotency-Key header.

    Request: 

    POST /api/v1/deals
Idempotency-Key: 7c6b5a4e-3d2f-1a0b-9c8d-7e6f5a4b3c2d

    How it works:

    1. FastAPI middleware checks Idempotency-Key header.
    2. If key is new → process request, store result in Redis (TTL: 24h).
    3. If key exists → return cached response (same status code + body).
    4. Key is scoped to the requesting user (different users with the same key are processed independently).

    Idempotency window: 24 hours. After that, a previously-used key is treated as new.

    The system: 1. Builds the MIME message 2. Sends via the connected provider's API (Gmail send, Graph sendMail) 3. Records the sent email in the emails table 4. Links to the deal and thread 5. Creates an activity entry

    File upload flow:

    Client uploads via multipart/form-data
  → FastAPI receives file (stream to temp location)
  → File metadata written to `files` table
  → Celery task queued:
    1. Upload file to R2
    2. Generate thumbnail (image files) → upload to R2
    3. Virus scan
    4. Update processing_status = 'ready'
  → Response returns immediately with file metadata + signed upload URL (direct-to-R2 for large files)

    For files >10MB, use direct-to-R2 upload flow: 

    POST /api/v1/files/upload-url   → Returns pre-signed R2 upload URL
PUT <r2-upload-url>             → Client uploads directly to R2
POST /api/v1/files/confirm      → Confirms upload, triggers processing

    Search Architecture

    User types query → FastAPI search endpoint
  → Clerk JWT auth (extracts tenant_id)
  → Proxy to Meilisearch with tenant token filter
  → Meilisearch returns results + facets
  → FastAPI enriches results (adds URLs, user names)
  → Returns to client (cached for 10s in Redis for identical queries)

    Search is at-most-30-seconds stale. Meilisearch indexes are updated via Celery tasks on entity create/update/delete. Redis caches identical queries for 10 seconds to handle rapid typing.

    17.3 Webhook Delivery

    Delivery flow:

    Event occurs (e.g., deal.stage_changed)
  → Event published to Redis pub/sub channel: 'webhook:outbound'
  → Celery task picks up the event
  → For each matching webhook endpoint:
     → POST to endpoint URL with HMAC-SHA256 signature header
     → Expect 2xx response within 5 seconds
     → On failure: retry with exponential backoff (1s, 5s, 30s, 5m, 30m, 6h)
     → After 6 failed attempts: mark webhook as 'failing', notify tenant admin

    Webhook payload: 

    POST https://customer-app.com/webhooks/crm
Content-Type: application/json
X-FrontierCRM-Signature: sha256=abc123def456
X-FrontierCRM-Event: deal.stage_changed
X-FrontierCRM-Delivery-Id: del_abc123
