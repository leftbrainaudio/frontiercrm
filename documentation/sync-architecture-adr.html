    1. Context

    FrontierCRM needs bidirectional sync between CRM data and external providers (Gmail, Microsoft Graph, Google Calendar, Outlook Calendar). The sync must handle:

    • Email sync: Read emails from providers, link to CRM contacts/deals, track read/unread status, send emails via connected provider
    • Calendar sync: Sync events, detect conflicts, expand recurring events, support free/busy lookups
    • Bidirectional changes: A deal linked to an email thread in CRM should be reflected. A calendar event created in Google Calendar should appear in CRM, and an event created in CRM should appear in the user's calendar.
    • Delta sync: Only sync what changed since last sync, not the entire mailbox
    • Conflict resolution: Handle the case where the same entity is modified in both CRM and the provider simultaneously

    Provider Landscape

    Provider Email API Calendar API Sync Method Complexity
    Gmail Gmail API v1 + Pub/Sub Google Calendar API v3 historyId / syncToken Medium-High
    Microsoft Graph Graph /me/messages/delta Graph /me/calendarView/delta deltaLink Medium
    IMAP/SMTP IMAP UID + CONDSTORE None (CalDAV for calendar) UIDVALIDITY/MODSEQ High
    CalDAV N/A CalDAV sync-collection ETag / sync-token Very High

    2. Decision Drivers

    # Driver Constraint
    1 Async everything Never block an HTTP request on third-party API calls. All sync runs via Celery workers.
    2 Provider-agnostic core The CRM should not depend on provider-specific code. Every provider has an adapter with the same interface.
    3 Delta-first sync Full sync only on initial connection or when delta cursors expire. Normal operation is incremental.
    4 Eventually consistent Sync latency of 30–60s is acceptable for the SMB segment. Not a financial trading system.
    5 Idempotent operations Running the same sync twice should produce the same result.
    6 Traceable conflicts Every conflict is logged with before/after snapshots for audit/debugging.
    7 OAuth lifecycle Token refresh failures are surfaced to users. Connections with expired tokens are paused, not silently broken.
    8 Tiered sync intervals Free plan: 120s polling. Starter: 60s. Growth: 30s + webhooks. Enterprise: real-time webhooks.

    3. Options Considered

    Option 1: Build In-House Sync Engine (Selected)

    Build a unified sync engine with adapter pattern — one adapter per provider.

    Cost: 6–10 weeks development (email + calendar for Graph and Gmail) Per-user cost: $0 (no middleware fees) Control: Full control over conflict resolution, data model, OAuth flow Provider coverage: Graph + Gmail Calendar + Google Calendar (80% of SMB). IMAP/CalDAV via middleware.

    Option 2: Sync Middleware (Nylas / Cronofy)

    Use a third-party sync middleware that abstracts multiple providers.

    Cost: $2–5/user/month (at 100 users: $200–500/mo) Time to market: 1–2 weeks Data privacy: Email bodies pass through third-party service Provider coverage: All providers out of the box Risk: Vendor lock-in, data residency concerns, adds latency

    Option 3: Hybrid (Selected Variation)

    Build native adapters for Graph and Google Calendar (covering 80%+). Use middleware for IMAP/CalDAV long-tail.

    Cost: 4–6 weeks for native adapters + $200/mo for middleware covering the remaining 20% Best of both: Full control over primary providers, rapid coverage of edge cases

    Recommendation: Option 3 (Hybrid). Build native Graph and Google Calendar adapters first. Add middleware for IMAP/CalDAV in Phase 2 when demand materializes.

    4. Decision: Unified Sync Engine with Adapter Pattern

    Why In-House for Primary Providers

    1. Data model control. Email body parsing, thread grouping, and CRM linking are core CRM features. Building them on top of a middleware limits what we can do.
    2. Cost. At $2–5/user/month, middleware costs exceed FrontierCRM's target margin (planning $29/user/mo). 100 users on middleware = $200–500/mo vs $0 for in-house.
    3. Latency. Every middleware hop adds 100–500ms. For sync operations that already have provider latency (100–1000ms), this cumulatively hurts.
    4. Data privacy. Email bodies contain sensitive customer data. Keeping them within our infra avoids another party having access.
    5. OAuth ownership. Managing OAuth tokens ourselves means we control refresh, expiry handling, and user-facing error messages.

    Provider Priority

    Phase Provider Sync Type Implementation
    P0 Microsoft Graph Email + Calendar Native adapter (build)
    P0 Google Calendar Calendar Native adapter (build)
    P1 Gmail API Email Native adapter (build)
    P2 IMAP/SMTP Email only Middleware (Nylas)
    P3 CalDAV Calendar (read-only) Middleware (Cronofy)

    Adapter Interface

    class SyncAdapter(ABC):
    """Abstract interface for all sync providers."""

    # ========== Email ==========

    @abstractmethod
    async def get_email_delta(
        self, cursor: dict | None
    ) -> DeltaResult[EmailMessage]: ...

    @abstractmethod
    async def send_email(
        self, message: OutgoingEmail
    ) -> EmailSendResult: ...

    @abstractmethod
    async def mark_read(
        self, provider_message_id: str, is_read: bool
    ) -> None: ...

    @abstractmethod
    async def get_initial_cursor(self) -> dict: ...

    # ========== Calendar ==========

    @abstractmethod
    async def get_event_delta(
        self, cursor: dict | None
    ) -> DeltaResult[CalendarEvent]: ...

    @abstractmethod
    async def create_event(
        self, event: CalendarEvent
    ) -> CalendarEvent: ...

    @abstractmethod
    async def update_event(
        self, provider_event_id: str, event: CalendarEvent
    ) -> CalendarEvent: ...

    @abstractmethod
    async def delete_event(
        self, provider_event_id: str
    ) -> None: ...

    @abstractmethod
    async def get_freebusy(
        self, time_min: datetime, time_max: datetime, users: list[str]
    ) -> FreeBusyResult: ...

    # ========== Lifecycle ==========

    @abstractmethod
    def validate_connection(
        self, access_token: str
    ) -> ConnectionStatus: ...

    @abstractmethod
    def refresh_token(
        self, refresh_token: str
    ) -> TokenRefreshResult: ...

    Sync Worker Flow (Detailed)

    sync_worker(connection_id, sync_type):
  1. Load connection + sync_state from DB
  2. Determine adapter class from connection.provider
  3. Initialize adapter with OAuth tokens
  4. If tokens expired → attempt refresh
     - If refresh fails → mark connection as 'expired', notify user, abort
  5. Load cursor from sync_state.cursor_data
  6. If no cursor (first sync) → full_sync:
     a. Call adapter.get_initial_cursor()
     b. Call adapter.get_email_delta(None) → full result
     c. Process all items (batch insert, create activities)
     d. Save cursor
  7. If cursor exists → delta_sync:
     a. Call adapter.get_email_delta(cursor)
     b. If full_resync_required → goto full_sync
     c. For each new/modified item:
        - Upsert into emails/events table
        - Update or create CRM link (contact matching by email)
        - Create activity entry in timeline
     d. For each deleted item:
        - Soft-delete in CRM (don't hard delete — user may want to review)
        - Create activity entry for deletion
     e. Save new cursor
  8. Update sync_state (last_delta_sync_at, total_synced_count, etc.)
  9. Update sync_connection (last_sync_at, last_sync_success = true)
  10. Handle rate limits:
      - If 429 received → respect Retry-After, schedule next sync with backoff
      - Increment error_count, handle consecutive failures

    6. Adapter Implementations

    6.1 Microsoft Graph Adapter

    Email delta sync: 

    GET /me/messages/delta?$select=id,subject,from,toRecipients,...
  → Response includes @odata.deltaLink
  → Store deltaLink in sync_state.cursor_data
  → Next call: GET {deltaLink}

    Calendar delta sync: 

    GET /me/calendarView/delta?startDateTime=2026-01-01T00:00:00Z&endDateTime=2027-01-01T00:00:00Z
  → Response includes @odata.deltaLink
  → Store deltaLink
  → Next call: GET {deltaLink}

    Key notes: - deltaLink works for ~14 days before requiring full resync - Use $select to minimize payload (don't fetch full body on list calls) - Batch individual message fetches when full body is needed - Differentiate Outlook.com (consumer) vs Office 365 (business) — feature detection via API

    6.2 Gmail Adapter

    Email delta sync: 

    1. Start sync:
   POST /gmail/v1/users/me/watch → { historyId, expiration }
   GET /gmail/v1/users/me/messages?maxResults=100&q=after:{timestamp}

2. Delta sync (after initial):
   GET /gmail/v1/users/me/history?startHistoryId={historyId}
   → Returns { history: [{ messagesAdded, messagesDeleted, labelsAdded, labelsRemoved }] }
   → Store latest historyId for next call

    Key notes: - historyId expires after ~7 days → fallback to full time-range sync - Pub/Sub watch renewal is optional (skip for MVP — use polling + history) - Need separate GCP Pub/Sub topic per user for push (costly at scale — polling is cheaper) - Fetch full message body via GET /gmail/v1/users/me/messages/{id}?format=full - Send via POST /gmail/v1/users/me/messages/send with RFC 2822 MIME body

    6.3 Google Calendar Adapter

    Calendar delta sync: 

    1. Initial sync:
   GET /calendar/v3/calendars/primary/events?singleEvents=true&timeMin=2026-01-01T00:00:00Z
   → Response includes nextSyncToken in nextPageToken field
   → Store syncToken

2. Delta sync:
   GET /calendar/v3/calendars/primary/events?syncToken={token}
   → Returns changed events + cancelled events (status: "cancelled")
   → On 410 Gone → full resync

    Key notes: - syncToken expires after ~30 days (undocumented, but empirical) - Use singleEvents=true for expanded recurring events (simpler than managing RRULE ourselves) - For two-way sync: events created in CRM get POST to Google Calendar; events modified in Google Calendar come via delta - Free/busy via POST /calendar/v3/freeBusy

    6.4 IMAP Adapter

    Email delta sync: 

    1. Initial sync:
   - Connect to IMAP with XOAUTH2
   - SELECT INBOX
   - Store UIDVALIDITY, UIDNEXT
   - FETCH UID 1:* (FLAGS BODY.PEEK[HEADER.FIELDS (DATE FROM SUBJECT MESSAGE-ID)])

2. Delta sync:
   - UID SEARCH UID UIDNEXT:* (new messages since last sync)
   - UID SEARCH MODSEQ {last_modseq} * (changed flags since last sync)
   - Store updated UIDNEXT, MODSEQ

    Key notes: - IMAP IDLE not used — doesn't scale beyond ~50 concurrent connections - Only sync inbox and sent folders (not spam, trash) - Body parsing done client-side (MIME extraction via Python's email stdlib) - Gmail IMAP quirks: labels map poorly to folders, All Mail contains everything

    7. Delta Sync Strategy

    Cursor Types by Provider

    Provider Email Cursor Calendar Cursor Expiry Full Resync Trigger
    Graph deltaLink URL deltaLink URL ~14 days 414 URI too long, 410 Gone
    Gmail historyId (integer) syncToken (string) ~7 days (email), ~30 days (calendar) 404 history not found, 410 Gone
    IMAP uidvalidity + uidnext + modseq N/A Permanent (uidvalidity changes on mailbox recreate) UIDVALIDITY changed
    CalDAV N/A sync-token (opaque token) Server-dependent sync-token expired

    Delta Sync Decision Tree

    Has existing cursor in sync_state?
  ├── Yes → Attempt delta sync
  │         ├── Success → Update cursor, mark success
  │         ├── 404/410/Expired cursor → Full resync
  │         └── 429 rate limit → Backoff, retry with Backoff-After
  └── No → Full sync (first connection or reset)
          └── Save initial cursor for next delta

    Full Resync Strategy

    Full resyncs are expensive (could be 10K+ emails for a 2-year mailbox). To mitigate:

    1. Throttle full resync to once per 24 hours per connection
    2. Yield between batches — after processing 100 items, check if there's more work to do
    3. Priority by recency — sync last 30 days first (high-value), backfill older emails in low-priority background queue
    4. Monitor — alert on full resyncs (indicates cursor expiry pattern → bug or provider issue)

    8. Conflict Resolution

    Conflict Types

    Type Scenario Resolution Strategy
    Concurrent edit Email marked read in CRM + read in Gmail simultaneously Last-write-wins (compare updated_at timestamps)
    Delete elsewhere Email deleted in Gmail but still in CRM CRM marks deleted — soft-delete in CRM, log in sync_conflicts
    Version mismatch Email body differs between CRM and provider Provider wins — trust external source of truth for email bodies
    Calendar time conflict Event time changed in both CRM and Google Calendar Last-write-wins + log conflict for manual review
    Event deleted in CRM User removes event from CRM that exists in Calendar Propagate deletion to Calendar (with confirmation flow)

    Resolution Rules

    1. EMAIL: Provider wins for body, headers, attachments.
   CRM wins for: is_read, is_starred, labels, CRM links (deal_id, contact_id).
   Rationale: Email content is from the provider. CRM metadata is from our system.

2. CALENDAR EVENT: Last-write-wins based on last_modified_at.
   If timestamps are within 60 seconds → flag for manual review.
   If provider updated_at > crm updated_at → use provider version.
   If crm updated_at > provider updated_at → use CRM version → push to provider.

3. CONTACT: CRM wins for CRM-specific fields (lifecycle_stage, owner, tags).
   Provider wins for: name, email, phone, job_title.
   Rationale: Contact enrichment from email signatures, calendar attendees.

4. DELETE: Both directions propagate.
   Delete in CRM → sync to provider (if appropriate, e.g., calendar event).
   Delete in provider → soft-delete in CRM (never hard-delete).

    Conflict Logging

    Every unresolved or auto-resolved conflict is recorded in sync_conflicts:

    {
  "entity_type": "event",
  "entity_id": "uuid",
  "external_id": "provider-event-123",
  "conflict_type": "concurrent_edit",
  "resolution_strategy": "last_write_wins",
  "crm_version": { "title": "Meeting with Acme", "start_at": "2026-06-28T14:00:00Z", "updated_at": "2026-06-28T13:00:00Z" },
  "provider_version": { "title": "Call with Acme", "start_at": "2026-06-28T15:00:00Z", "updated_at": "2026-06-28T13:30:00Z" },
  "resolved_data": { "title": "Call with Acme", "start_at": "2026-06-28T15:00:00Z" }
}

    9. Real-Time Architecture

    Real-time updates in FrontierCRM are handled through two mechanisms:

    1. WebSocket connections — for live updates to the UI (pipeline changes, notifications)
    2. Redis Pub/Sub — the backbone connecting application events to WebSocket broadcast

    Architecture Diagram

    ┌─────────────┐     ┌─────────────────────────────────────────┐
│   Client A   │     │  FastAPI Application                     │

│   Client B   │◄────│  │  - Channel subscription mgmt     │    │

                    │             │                             │
                    │  ┌──────────▼───────────────────────┐    │
                    │  │   Redis Pub/Sub Subscriber        │    │
                    │  │   Subscribes to:                  │    │

    Event → Pub/Sub → WebSocket Flow

    1. Event occurs (e.g., user moves deal to new stage via PATCH /deals/{id}/move)
2. FastAPI handler:
   a. Performs the database update
   b. Creates activity record
   c. Publishes to Redis Pub/Sub: `ws:tenant:{tenant_id}:pipeline`
      Payload: { type: "deal.stage_changed", deal_id: "...", stage_id: "...", ... }
3. Redis Pub/Sub subscriber (running in the FastAPI process):
   a. Receives the message
   b. Looks up all WebSocket connections for this tenant
   c. For each connection, checks:
      - Is the user authorized to see this change? (RBAC check)
      - Is the user subscribed to this channel?
   d. Sends JSON message over WebSocket
4. Client receives message, updates UI (TanStack Query invalidation or direct state update)

    WebSocket Connection Management

    WebSocket URL: wss://api.frontiercrm.com/api/v1/ws

Connection establishment:
1. Client connects with `?token={clerk_jwt}` query parameter
2. FastAPI validates JWT (same middleware as REST)
3. Connection registered in WS connection registry (in-memory dict, keyed by tenant_id)
4. Client sends subscription request:
   {
     "type": "subscribe",
     "channels": ["pipeline", "notifications", "deal:uuid"]
   }
5. Acknowledged:
   {
     "type": "subscribed",
     "channels": ["pipeline", "notifications", "deal:uuid"]
   }

    Connection Registry

    The WebSocket connection registry is in-process (not Redis-backed) because:

    • With Cloud Run auto-scaling, WebSocket connections are sticky to a specific instance
    • Cross-instance broadcast happens via Redis Pub/Sub (every instance subscribes to the same channels)
    • If a user is connected to instance A, and a deal update comes in on instance B, the message flows: B → Redis → A → Client
    • This works because Cloud Run supports WebSocket connections (HTTP/1.1 upgrade) with a 60-minute timeout per connection; clients reconnect on timeout

    Important limitation: Cloud Run's WebSocket support has a 60-minute idle timeout. Clients must implement automatic reconnection. For Phase 2+, consider dedicated WebSocket infrastructure (Socket.io-compatible service or a purpose-built WS server).

    10. WebSocket Channels

    Channel Scope Payload Purpose
    pipeline Tenant { type, deal_id, stage_id, pipeline_id } Real-time kanban board updates
    notifications Tenant { type, title, body, action_url } Toast/notification stream
    deal:{deal_id} Entity { type, field, old_value, new_value } Collaborative deal editing
    sync:status User { connection_id, status, progress } Sync status updates
    user:{user_id} Personal { type, ... } Private messages (task assigned, @mention)

    Channel Authorization

    Channel Who Can Subscribe Notes
    pipeline All tenant users Broadcast all pipeline changes
    notifications All tenant users Filtered server-side per user
    deal:{deal_id} Users with deal access RBAC check on subscription
    sync:status Only the connection owner One-to-one
    user:{user_id} Only the user One-to-one

    11. Notifications Architecture

    Notification Types

    Type Trigger Delivery Frequency
    Deal stage changed Deal moved WS + Push + Email digest Immediate
    Task assigned Task created/assigned WS + Push + Email Immediate
    @mention in note Note body contains @username WS + Push Immediate
    Email matched to deal Inbound email linked to deal WS + Push + Email Immediate
    Sync error Connection error Push + Email Immediate
    Daily digest Scheduled Email Daily (configurable)
    Weekly report Scheduled Email Weekly (configurable)

    Notification Delivery Flow

    Notification triggered
  → Published to Redis `ws:tenant:{id}:notifications`
  → WebSocket subscriber delivers to connected clients (in-app toast/notification badge)
  → If user not connected (or notification is "push" type):
    → Queue push notification via Web Push API (PWA) or native push (when RN app exists)
  → If notification is "email" type:
    → Queue email via Resend

    Push Notification (PWA)

    POST /api/v1/push/subscribe
Body: { endpoint, keys: { p256dh, auth } }

→ Store subscription in push_subscriptions table

On notification:
→ Fetch user's push subscriptions
→ Send via Web Push API (VAPID keys)
→ Handle 410 Gone (subscription expired) → remove subscription

    12. Search Architecture

    Index Strategy

    Meilisearch indexes are updated via Celery workers triggered by entity create/update/delete events.

    Entity changed (Contact created, Deal updated, Email synced)
  → Publish to Redis pub/sub: 'search:index'
  → Celery worker picks up the event
  → Calls Meilisearch client to add/update/delete document
  → Documents include tenant_id for multi-tenant filtering

    Index Configuration

    # Meilisearch index settings (applied on startup via migration)
INDEX_SETTINGS = {
    "contacts": {
        "filterableAttributes": ["tenant_id", "lifecycle_stage", "owner_id", "tags"],
        "sortableAttributes": ["created_at", "last_name"],

    Search Query Flow

    Client: GET /api/v1/search?q=acme&filter[entity]=contact,deal
  → FastAPI search endpoint
  → Extract tenant_id from JWT
  → Build Meilisearch query:
     Multi-search across contacts, deals indexes
     Filter: tenant_id == current_tenant_id
     Limit: 20 per index
     Facets: [entity, stage, owner_id]
  → Merge results, enrich with CRM context (owner names, URLs)
  → Return { results, facets }

    PostgreSQL FTS Fallback

    If Meilisearch is unavailable, the search endpoint falls back to PostgreSQL full-text search:

    SELECT id, first_name || ' ' || last_name AS name, company,
       ts_rank(search_vector, plainto_tsquery('english', $1)) AS rank
FROM contacts
WHERE tenant_id = $2
  AND search_vector @@ plainto_tsquery('english', $1)
  AND deleted_at IS NULL
ORDER BY rank DESC
LIMIT 20

    How It Works

    Primary path: Meilisearch
  → 10–50ms response, typo-tolerant, faceted
Fallback path: PostgreSQL FTS
  → 50–200ms response, no typo tolerance, no facets
Health check: Cron job pings Meilisearch every 30s
  → If unavailable, set feature flag: use_postgres_search = true

    13. File & Attachment Storage Architecture

    Storage Decision: Cloudflare R2

    As decided in ADR-010, Cloudflare R2 is the file storage backend (S3-compatible, zero egress fees).

    Upload Flow

    Small files (<10MB) — Proxy upload:
  Multipart POST /api/v1/files/upload
    → FastAPI receives file stream
    → Writes to temp location
    → Creates file record (processing_status = 'pending')
    → Queues Celery task:
      1. Upload to R2 (tenant_id/uuid/filename)
      2. Virus scan (ClamAV or API-based)
      3. Thumbnail generation (images — resize to 200x200, 800x800)
      4. Generate signed URL (24h TTL)
      5. Update file record: processing_status = 'ready'
    → Returns file metadata + placeholder

Large files (>10MB) — Direct-to-R2 upload:
  POST /api/v1/files/upload-url
    → Returns { upload_url, file_id, expires_at }
  Client uploads PUT to R2 upload_url
  POST /api/v1/files/confirm { file_id, storage_key }
    → Triggers processing (virus scan, thumbnail)

    Signed URL Strategy

    Use Case TTL Notes
    Direct download 1 hour Standard download link
    Email preview 24 hours Inline image display
    Attachment download (bulk) 1 hour Zip archive download
    Thumbnail 7 days Cached previews, longer TTL
    Profile avatar 30 days Public profile images (can be public)

    Storage Key Structure

    {tenant_id}/{entity_type}/{entity_id}/{uuid}_{original_filename}
  → e.g., "abc123/contact/def456/a1b2c3_avatar.jpg"
  → e.g., "abc123/deal/789012/a1b2c3_proposal.pdf"
  → e.g., "abc123/email/345678/a1b2c3_attachment.docx"

    Processing Pipeline

    File uploaded → Celery task queued:

1. VIRUS SCAN (Celery worker):
   - For files > 100MB: skip scan, mark as 'skipped'
   - For known-safe types (.txt, .csv, .md): quick check
   - For all others: ClamAV scan
   - If infected: mark file processing_status = 'failed', notify uploader

2. THUMBNAIL GENERATION (Celery worker):
   - For images: generate 200x200 and 800x800 JPEG thumbnails
   - For PDFs: generate first-page thumbnail (via PyMuPDF)
   - For videos: generate frame at 00:01 (via ffmpeg)
   - Upload thumbnails to R2 with key: {storage_key}/thumb_{size}.jpg

3. SEARCH INDEX UPDATE (Celery worker):
   - Add file metadata to Meilisearch notes index (filename, mime_type)
   - Future: OCR for searchable PDF content (Phase 2)

    14. Consequences

    Positive

    1. Provider-agnostic core — Adding a new provider (Yahoo Mail, ProtonMail, iCloud) means writing one adapter class. The sync engine, conflict resolution, and data model don't change.
    2. Delta sync efficiency — After the initial full sync, most operations process 0–5 changed items per polling interval. Bandwidth and API quota usage are minimal.
    3. Conflict traceability — Every conflict is logged with before/after snapshots. Debugging sync issues is straightforward.
    4. Tiered sync — The same engine powers all plans. Free users get fewer polling intervals and no webhooks; Enterprise users get real-time.
    5. Real-time without extra infra — Redis Pub/Sub + in-process WebSocket manager provides real-time updates without a dedicated service.

    Negative

    1. OAuth maintenance burden — We manage token refresh, expiry, and error handling for every provider. A token refresh bug affects all sync for that user.
    2. WebSocket state on Cloud Run — WebSocket connections are limited by Cloud Run's 60-minute timeout and instance-based connection registry. Clients must handle reconnection.
    3. Full resync cost — If a delta cursor expires (e.g., user hasn't opened CRM in 30 days), sync will require a full mailbox pull. Mitigated by the priority-by-recent-customer approach.
    4. IMAP complexity — IMAP adapter is the most complex to build correctly. Multi-part MIME parsing, UIDVALIDITY tracking, and flag synchronization are error-prone.
    5. Meilisearch sync latency — There's a ~5–30 second lag between a CRM change and it appearing in search results. Acceptable for SMB but worth noting.

    15. Risks & Mitigations

    Risk Likelihood Impact Mitigation
    OAuth token refresh fails silently Medium High Alerting on failed refresh, user-facing "Reconnect" prompt, automatic connection pausing
    Gmail API daily quota exceeded at 200+ users Medium Medium Per-user quota tracking, staggered sync schedules, IMAP fallback for heavy users
    WebSocket cloud-run timeout drops client connection High Medium Client-side auto-reconnect with exponential backoff, last-known-state rehydration
    Meilisearch goes down Low Medium Transparent fallback to PostgreSQL FTS, health-check-based feature flag
    Calendar event timezone mismatch Medium Medium Always store in UTC, convert on display, log timezone mismatch warnings
    Delta cursor expiry causes full resync Medium Low Alert on cursor expiry pattern; implement priority-by-recent-customer backfill
    IMAP Gmail label mapping loss High Low Store both IMAP folder and Gmail label IDs, document lossy mapping in UI
    Concurrent sync for same user from multiple instances Low Medium Redis lock per connection_id during sync; re-queue on lock conflict
    Provider API deprecation (e.g., Graph beta endpoint removed) Low High Adapter version pinning, integration tests against v1.0 endpoints, deprecation monitoring
    Large email attachments consume disk space Medium Low Direct-to-R2 upload for >10MB, no temp storage on application server
