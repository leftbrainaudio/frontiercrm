    FrontierCRM — Social & Analytics Integration Specifications

    Target market: SMBs, 10–200 employees
    Scope: Social APIs (LinkedIn, Twitter/X, Meta), Workflow Automation (Zapier, Make), Analytics (GA4, Mixpanel, Tableau)
    Format per entry: Integration pattern | Auth type | Sync complexity | Common pain points | Strategic recommendation for FrontierCRM

    1. LinkedIn API

    Overview

    LinkedIn API v2 is heavily restricted. Post-2015 changes gutted most profile-access endpoints; what remains is oriented toward marketing, advertising, and verified "Community" access. For a CRM, LinkedIn is useful for social posting, company page analytics, and lead-gen forms, but not for profile enrichment or prospecting at scale.

    Key Details

    Aspect Detail
    API Version v2 (granular REST endpoints, no monolithic GraphQL)
    Base URL https://api.linkedin.com/v2/
    Auth OAuth 2.0 (Authorization Code flow)
    Token Expiry Access tokens: 2h (short-lived); Refresh tokens: available via offline_access scope, duration set by LinkedIn
    Rate Limits 100 calls/day per user per app (very restrictive for most endpoints). Marketing API is higher but requires contract.

    Available Scopes (Realistic)

    Scope What It Actually Does CRM Usefulness
    openid Sub (member ID), name ✅ Identity linking
    profile Given name, family name, vanity name (not email, not photo) ⚠️ Limited
    email Verified email address ✅ Critical for CRM
    w_member_social Post as the authenticated user ✅ Social publishing
    r_organization_social Read org posts + analytics ✅ Company page analytics
    w_organization_social Post as a company page ✅ Social publishing
    rw_organization_admin Manage company pages (requires Business tier) ⚠️ Paid gate
    r_ads_leadgen Read lead-gen form submissions ✅ Lead capture

    What's Actually Available for CRM

    • Basic Profile — name, vanity URL, headline. No email (separate scope), no full work history (requires Legacy API), no profile photo URL, no list of connections.
    • Share Content — post text/image/video as user or company page. Works well.
    • Company Page Analytics — impressions, clicks, engagement metrics. Requires r_organization_social.
    • Lead Gen Forms — read form submissions from LinkedIn ads. High-value for CRM.
    • Conversations/Messaging — NOT available via public API for non-Messaging Partner apps.

    Common Pain Points

    1. No profile photo - profile scope returns only text fields; photos require Marketing Developer Platform approval (rare).
    2. No connection list - eliminated in 2015; CRM cannot show mutual connections or 2nd-degree networks.
    3. No full work history - r_fullprofile was removed. Only current position + headline.
    4. Share API requires manual scope grant - users must accept w_member_social scope. Cannot auto-publish without explicit authorization via OAuth popup.
    5. Company page posting requires admin approval - the page admin must grant the app access via page settings.
    6. Rate limits are brutal - 100 calls/day/user for most v2 endpoints; effectively limits any polling/recurring sync.
    7. API changes with little notice - LinkedIn deprecates endpoints frequently; community developer forums report sudden 403 errors.
    8. App Review required for almost everything - Marketing Developer Platform application can take weeks.

    Integration Pattern

    [FrontierCRM] <- OAuth 2.0 -> [LinkedIn API v2]
       |
  [User consents scopes via OAuth popup]
  [Manual page admin grant for company posting]

    Sync Complexity: Low (read-only profile/analytics) to Medium (social posting with media upload) Sync Type: On-demand + polling (subject to rate limit) Webhooks: No public webhooks available

    Strategic Recommendation for FrontierCRM

    Use Case Recommend? Why
    Social posting to LinkedIn Yes Straightforward, high value for SMB marketing.
    Company page analytics display Yes Data is useful (impressions, clicks, followers).
    Lead-gen form capture Yes Good lead source automation.
    Profile enrichment (photo, history, connections) No Not technically possible.
    Automated contact matching/linking Limited Can match via email + LinkedIn sub id only.

    Strategy: Offer LinkedIn as a social publishing + lead capture integration only. Do NOT market it as a profile enrichment or prospecting tool. Use email scope for identity reconciliation. Keep sync frequency low (daily instead of real-time) to stay within rate limit guardrails.

    2. Twitter/X API

    Overview

    Twitter API v2 (launched 2020, deprecated v1.1 in 2023) is the current standard. It's project-based with tiered access (Free, Basic, Pro, Enterprise). OAuth 2.0 is now the recommended auth path for most endpoints, though OAuth 1.0a remains for user-context write operations.

    Key Details

    Aspect Detail
    API Version v2 (v1.1 deprecated April 2023)
    Base URL https://api.twitter.com/2/
    Auth OAuth 2.0 (PKCE for read), OAuth 1.0a (user write actions)
    Token Expiry OAuth 2.0: 2h access + refresh tokens; OAuth 1.0a: doesn't expire
    Pricing Free: 500 posts/mo, Basic: $100/mo (3K posts, 10K tweets read), Pro: $5K/mo (300K posts, 1M tweets read), Enterprise: custom

    Available Endpoints (Relevant to CRM)

    Endpoint v2 Path CRM Use
    User lookup /users/:id / /users/by/username/:username Identity matching
    Tweet lookup /tweets/:id Read individual posts
    Timeline /users/:id/tweets / /users/:id/mentions Monitor user activity
    Search /tweets/search/recent (last 7d) / /tweets/search/all (full archive, Enterprise) Social listening
    Post tweet /tweets (OAuth 1.0a) Social publishing
    Media upload https://upload.twitter.com/1.1/media/upload.json (still v1.1) Image/video attach
    DM endpoints /dm_conversations/... Messaging -- requires separate approval
    Filtered stream /tweets/search/stream / /tweets/search/stream/rules Real-time monitoring
    Compliance /tweets/compliance/stream Data deletion compliance

    Rate Limits (Per 15-min Window)

    Plan Posts Reads (tweets)
    Free 50 100
    Basic ($100/mo) 200 300
    Pro ($5K/mo) 1,500 3,000
    Enterprise Custom Custom

    Common Pain Points

    1. Rate limits are aggressively tight - Free tier (100 tweets/15min) is unusable for any batch sync. Even Basic (300/15min) limits multi-user CRM workflows.
    2. DM endpoints require additional approval - separate application process, approved only for "use cases that enhance the conversation experience."
    3. OAuth 1.0a still required for write - OAuth 2.0 only supports read operations for user context. Write (posting) still needs the older OAuth 1.0a flow.
    4. Media upload still on v1.1 - No v2 media endpoint exists. Must call deprecated v1.1 endpoint with OAuth 1.0a.
    5. Filtered stream complexity - Rules syntax (punctuation, group/operator precedence) is error-prone.
    6. Pricing jumped dramatically - In 2023, Twitter went from free to $100/mo (Basic) minimum for any useful access.
    7. API documentation churn - Post-acquisition documentation has been inconsistent.
    8. Search recency window - Standard search is last 7 days only. Full archive requires Enterprise.

    Integration Pattern

    [FrontierCRM] <- OAuth 2.0 (read) + OAuth 1.0a (write) -> [Twitter API v2]
       |
  [OAuth 1.0a tokens don't expire]
  [OAuth 2.0 PKCE for read-only user context]

    Sync Complexity: Low-Medium Sync Type: Polling for tweets/timelines; streaming connection for filtered stream Webhooks: No webhooks

    Strategic Recommendation for FrontierCRM

    Use Case Recommend? Why
    Social posting (tweet scheduling) Yes Core CRM social feature, straightforward.
    Social listening (keyword search) Basic plan only Useful but $100/mo minimum for any real volume.
    DM automation / chatbot No DM approval is hard to get.
    Profile enrichment No Limited data; Twitter bios are unstructured.
    Real-time monitoring Pro plan+ Only for highest-tier customers.

    Strategy: Offer Twitter/X as a social publishing integration only, with tiered access reflecting API pricing. Bundle Basic plan cost ($100/mo) into a premium tier. Use OAuth 1.0a for write to avoid token refresh complexity.

    3. Facebook & Instagram (Meta Graph API)

    Overview

    The Meta Graph API v19+ provides access to Facebook Pages, Instagram Business accounts, lead ads, and messaging. It is the most powerful social integration for CRM due to Lead Ads -- users submit info inside FB/IG without leaving the platform, and webhooks deliver it to your CRM.

    Key Details

    Aspect Detail
    API Version Graph API v19+ (v20 current)
    Base URL https://graph.facebook.com/v20.0/
    Auth OAuth 2.0 (FB Login, Page Access Tokens, App Access Tokens, System User Tokens)
    Token Expiry User tokens: 60d; Page tokens: 60d (auto-extendable); App tokens: never
    Rate Limits 200 calls/user/hour, 4800 calls/app/day. Dynamic (based on past usage).

    Critical Endpoints for CRM

    Endpoint Graph Path CRM Value
    Page info /{page-id}?fields=name,picture,link,followers_count Basic page data
    Page posts /{page-id}/posts Read/schedule posts
    Page feed /{page-id}/feed Full activity stream
    Conversations /{page-id}/conversations Page inbox messages
    Messages /{conversation-id}/messages Individual messages
    Lead ads /{page-id}/leadgen_forms CRITICAL
    Lead data /{leadgen-id}/ Individual lead submission data
    Webhook /{app-id}/subscriptions Real-time lead delivery
    Instagram /{ig-user-id}/media Instagram media/posts
    Instagram messaging /{ig-user-id}/conversations IG DMs

    App Review Requirements

    Feature Review Required?
    Page read No (Public)
    Page manage Yes
    Lead ads read Yes
    Messaging Yes
    Instagram Basic Yes
    Ads Management Yes
    Webhooks No (for subscribed fields)

    Lead Ads (CRM Killer Feature)

    User fills lead form in FB/IG
       |
Meta sends POST to CRM webhook URL
       |
CRM receives: name, email, phone, custom questions
       |
CRM creates or updates contact record

    Common Pain Points

    1. App Review gate is real - Video screencast, privacy policy, business justification required. Review takes days to weeks.
    2. Token expiry management - User/Page tokens last 60 days. Auto-extend by calling endpoint before expiry.
    3. Page-scoped IDs (PSIDs) - Different user IDs per page. Cross-page dedup requires email/phone matching.
    4. App-scoped IDs (ASID) - Every app gets its own namespace of user IDs.
    5. Rate limits are opaque - Dynamic rate limits from past 24h usage. Monitor X-Business-Use-Case-Usage header.
    6. Lead ad duplicates - Meta may resend; CRM must deduplicate on lead ID.
    7. Instagram Basic Display API vs Graph API - Two different APIs, confusing.
    8. Platform policy changes - Auto-commenting, auto-liking are forbidden.

    Integration Pattern

    [FrontierCRM] <- OAuth 2.0 -> [Meta Graph API v20]
       |                   |
  [Page Token]       [Webhook endpoint]
       |                   |
  [Read/write]       [Receive leads in real-time]

    Sync Complexity: Medium Sync Type: Polling (posts, conversations) + Webhooks (lead ads) Webhooks: CRITICAL - Lead Ads webhook is the primary path

    Strategic Recommendation for FrontierCRM

    Use Case Recommend? Why
    Lead Ads capture Critical Best lead source for SMBs. Top priority.
    Page conversation inbox Yes Unifies Facebook messages in CRM.
    Social posting (page) Yes Schedule/publish to Facebook + Instagram.
    Instagram lead capture Yes Same lead ads system.
    Ads management Niche Only for SMBs running ads through CRM.
    Profile enrichment No PSID/ASID fragmentation blocks this.

    Strategy: Make Lead Ads the headline integration. Invest in: App Review readiness, token auto-refresh every 50 days, lead deduplication on leadgen_id + ad_id, webhook reliability with exponential backoff. Budget 2-4 weeks for initial approval.

    4. Zapier

    Overview

    Zapier connects 7,000+ apps via Zaps. For CRM targeting SMBs, Zapier integration is table stakes. Zapier handles auth, polling, rate limiting, error handling; FrontierCRM just needs a REST API + webhooks.

    Key Details

    Aspect Detail
    Platform zapier.com
    Integration Model Zapier Platform (private or public integration)
    Integration Types Trigger (polling/REST hook), Action (REST write), Search (REST read)
    Auth Supported API Key, Basic Auth, OAuth 2.0, Digest Auth, Session Auth
    Rate Limits ~3 req/s per integration
    Partner Pricing Free to build; rev share to publish

    How Triggers & Actions Work

    Polling Triggers

    GET /api/zapier/triggers/new-contact?since={ISO_timestamp} returns items sorted by created_at desc.

    REST Hook Triggers (Preferred)

    FrontierCRM posts to Zapier webhook URL when events occur. Requires subscription registry.

    Actions

    POST /api/zapier/actions/create-contact - CRUD against FrontierCRM API.

    When Zapier Is Enough vs. Native Integration

    Scenario Zapier Native
    Connect 2+ apps Perfect Overkill
    High-volume sync (1000s/hr) Latency too high Yes
    Real-time sub-second 1-15min polling Webhooks
    Complex multi-step logic Paid plan needed Custom code
    Bi-directional sync No Full sync engine
    Cost-conscious SMB $19.99+/mo extra Included

    Common Pain Points

    1. Latency - Polling every 1-15 minutes.
    2. Cost at scale - 10 Zaps = $200-2000/mo for customer.
    3. Basic error handling - 8 retries over 5h, then pause.
    4. Limited data transformation - Formatter only does basic ops.
    5. Rate limit collisions - All Zaps from same IP pool.
    6. No bidirectional sync - Requires 2 Zaps, risk of loops.
    7. Painful debugging - Generic error messages.
    8. Manual field mapping - Clunky UI.
    9. OAuth token refresh - Must handle correctly or Zaps break silently.

    Integration Pattern

    [FrontierCRM REST API] <-> [Zapier Platform]

    Sync Complexity: Low Sync Type: Polling or Webhook (preferred) Webhooks: FrontierCRM sends events to Zapier

    Strategic Recommendation

    Build Zapier integration on day 1. Implementation: 1. REST API foundation (CRUD, paginated, timestamp-filtered) 2. Triggers: New Contact, New Deal, New Activity, Updated Contact/Deal 3. Actions: Create Contact/Deal/Task/Note, Update Contact/Deal, Find Contact by Email 4. Webhook support: POST /api/zapier/hooks 5. OAuth 2.0 with PKCE + refresh tokens

    Do not replace native integrations with Zapier for high-value connectors (Mailchimp, QuickBooks).

    5. Make (formerly Integromat)

    Overview

    Make is Zapier's competitor with more power for less money. Visual scenario builder with multi-step workflows, data transformation, router logic. Same integration model as Zapier - expose REST API + webhooks.

    Key Details

    Aspect Detail
    Former Name Integromat
    Platform make.com
    Auth Supported API Key, Basic Auth, OAuth 2.0, Digest Auth
    Data Transformation Built-in JSON/XML/text parse, regex, iterators, data store
    Pricing Free (1K ops/mo) -> $9/mo (10K) -> $16/mo (15K) -> $29/mo (25K)

    Make vs. Zapier

    Feature Zapier Make
    Visual editor Linear chain Canvas with branching, merging, loops
    Data transformation Formatter (basic) Aggregate, array, text parser, router
    Error handling 8 retries, pause Per-module handler, continue/rollback
    Webhooks REST hook trigger Dedicated module + custom JSON
    Data stores Not available Built-in key-value
    HTTP module Only via Webhooks Custom HTTP requests
    Price 10K ops $19.99/mo $9/mo
    App directory 7,000+ ~2,000+

    Common Pain Points

    1. Learning curve - Canvases intimidate less-technical users.
    2. Debugging complex scenarios - Hard to troubleshoot multi-branch.
    3. Data store limits - 5-100MB per plan.
    4. Fewer apps than Zapier (~2K vs 7K).
    5. Execution timeouts - 40 min default.
    6. Webhook payload - 5MB max on standard plans.
    7. Operations counting - Complex scenarios burn through ops.

    Integration Pattern

    [FrontierCRM REST API] <-> [Make Platform]

    Sync Complexity: Low (same API as Zapier) Webhooks: Make's webhook module is excellent

    Strategic Recommendation

    Launch Zapier and Make simultaneously - they consume the same REST API. Write Make-specific docs with scenario examples (routers, iterators, data stores). Message: "Connect to 7K+ apps via Zapier" (masses) vs "Build sophisticated automation with Make" (power users).

    6. Google Analytics 4 (GA4)

    Overview

    GA4 uses an event-based data model (vs UA's session/pageview). More complex for programmatic access but more powerful for custom reporting.

    Key Details

    Aspect Detail
    API Version Data API v1, Admin API v1, Real-time API v1
    Base URL https://analyticsdata.googleapis.com/v1beta/
    Auth OAuth 2.0 or Service Account (recommended)
    Rate Limits 50K req/day/property (free). Real-time: 300 req/60s.

    Key Differences from UA

    Aspect UA GA4
    Data model Session+Pageview Events+Parameters
    Property structure Account->Property->View Account->Property (no views)
    Custom dimensions Index-based Named parameters
    Sampling Heavily sampled No sampling
    PII exposure Possible Strictly prohibited
    Real-time Last 5 min Up to 30 min

    Common Pain Points

    1. Complexity vs UA - Full rewrite. Protobuf-style JSON with Metric/Dimension objects.
    2. No cross-property queries - Must query each property separately.
    3. No PII - Cannot join GA4 events to CRM contacts on identity.
    4. Daily quotas hit fast - < 35 reports/min hits 50K cap.
    5. Manual dimension mapping - Users must map event parameters to CRM fields.
    6. Service Account setup is multi-step - Confusing for SMBs.
    7. Data freshness - Standard reports have ~1-2h latency.

    Integration Pattern

    [FrontierCRM] <- Service Account (JWT) -> [GA4 Data API v1]

    Sync Complexity: Medium Webhooks: No webhooks

    Strategic Recommendation

    GA4 is table stakes but not a differentiator. Use Service Account auth. Focus on aggregate campaign/UTM analytics in CRM dashboards. Don't try to match GA4 users to CRM contacts - embed FrontierCRM's own event tracking for behavioral data. Implement schema caching from GA4 metadata endpoint.

    7. Mixpanel

    Overview

    Mixpanel is product analytics for event-based user behavior. Enables lead scoring, product usage insights, and cohort analysis. JQL is powerful but has a learning curve.

    Key Details

    Aspect Detail
    API Version v2 (REST) + JQL
    Base URLs REST: https://mixpanel.com/api/2/; Engage: https://api.mixpanel.com/engage
    Auth Service Token (API Secret, never expires) - preferred
    Rate Limits REST: 10-50 req/s; Export: 2 req/s; JQL: 2 concurrent

    Available Endpoints

    Endpoint Path CRM Use
    Events (query) /api/2/events/ Event metadata
    Events (export) /api/mixpanel.com/export Raw event data
    Insights /api/2/insights/ Saved insights
    Engage (People) /api/2/engage User profiles
    Create profile /api/mixpanel.com/engage#profile-set Upsert profiles
    Cohorts /api/2/cohorts/ List cohorts
    Cohort members /api/2/cohorts/:id/members Members
    Funnels /api/2/funnels/ Funnel analysis
    JQL Query /api/2.0/jql Custom queries

    Common Pain Points

    1. JQL learning curve - Unique syntax. No SQL familiarity helps.
    2. Two APIs - REST for reports, JQL for custom queries.
    3. Engage API is separate - Different base URL and auth nuances.
    4. Upsert, not sync - Mixpanel profiles are event-driven; CRM manages consistency.
    5. Low rate limits for batch - Export 2 req/s. JQL 2 concurrent.
    6. No event webhooks - No real-time event stream.
    7. Retention varies by plan - Free: 5yr; Growth/Enterprise: unlimited.
    8. Cohort queries are slow - Large cohorts take 10+ seconds.
    9. Distinct ID management - Must map distinct_id to CRM contact ID.
    10. Historical import schema validation - Rejects unknown properties.

    Integration Pattern

    [FrontierCRM] <- Service Token -> [Mixpanel REST API v2 + JQL]

    Sync Complexity: Medium-High Webhooks: Limited (alerts only)

    Strategic Recommendation

    Mixpanel is a differentiator for FrontierCRM. While GA4 is generic website analytics, Mixpanel provides product usage signals for lead scoring and health monitoring.

    1. Service Token auth - simplest integration path
    2. REST API for common queries, JQL for power users
    3. Periodic cohort sync - tag CRM contacts as cohort members
    4. Store distinct_id as custom field on contacts
    5. Cache results for 1-6 hours
    6. Document JQL examples for common SMB use cases

    Position FrontierCRM as more sophisticated for product-led growth SMBs (SaaS, marketplace, e-commerce).

    8. Tableau

    Overview

    Tableau is enterprise BI. For SMBs 10-200 employees, deep Tableau integration is low value - expensive ($75/user/mo), complex, most don't use it.

    Key Details

    Aspect Detail
    API Version REST API v3.16+
    Auth PAT (Personal Access Token) - recommended
    Token Expiry PAT: never; Session: 120 min
    Rate Limits No hard limits; ~5 req/s recommended
    On-prem vs Cloud Different base URLs, same REST API

    Available Endpoints

    Endpoint Path CRM Use
    Workbooks /sites/{id}/workbooks List/publish
    Views /sites/{id}/views List dashboards
    Subscriptions /sites/{id}/subscriptions Schedule emails
    Webhooks /sites/{id}/webhooks Event listeners
    Users/Groups /sites/{id}/users User management

    Auth Flow (PAT)

    1. User generates PAT in Tableau UI
    2. User enters PAT name+secret into FrontierCRM
    3. POST /auth/signin with PAT -> session token
    4. Session token in X-Tableau-Auth header
    5. Re-authenticate when session expires (~120 min)

    Common Pain Points

    1. Enterprise-focused - AD/LDAP, SAML, Kerberos expected.
    2. Heavy auth setup - 2-step warmup for every API chain.
    3. Expensive - Creator $75/user/mo. Most SMBs can't justify.
    4. Self-hosted vs Cloud divergence - CRM must handle both.
    5. Multiple ID types - Content URL, repo URL, API ID.
    6. Site ID required - Must discover and store.
    7. GraphQL experimental - REST only for production.
    8. Admin-configured rate limits - No warning until 429.
    9. Embed complexity - iframe + signed ticket via Trusted Authentication.
    10. pageNumber/pageSize pagination - Slow for large sets.

    Integration Pattern

    [FrontierCRM] <- PAT -> [Tableau REST API v3.16+]

    Sync Complexity: High Webhooks: Available

    Strategic Recommendation

    Do NOT invest heavily in Tableau. Fewer than 5% of SMBs 10-200 use it. Prioritize Power BI instead (included in M365 Business Premium, reaches 10-20x more SMBs).

    Minimum viable: PAT auth only, view list + iframe embed, webhook for workbook changes. Don't build subscriptions, data sources, or user management.

    Summary Comparison Matrix

    Integration Auth Complexity Webhooks Key CRM Value SMB Fit Priority
    LinkedIn OAuth 2.0 Low-Med No Social posting, lead-gen Good High
    Twitter/X OAuth 1.0a+2.0 Low-Med No (stream) Scheduling, listening Cost barrier Medium
    Meta (FB/IG) OAuth 2.0 Medium Lead Ads Lead capture, inbox Critical Highest
    Zapier OAuth 2.0 Low Consumer 7K+ app connectivity Table stakes Critical
    Make OAuth 2.0 Low Consumer Advanced automation Good High (with Zapier)
    GA4 Service Account Medium No Website analytics Expected High
    Mixpanel API Secret Med-High Limited Product usage scoring Differentiator Medium
    Tableau PAT High Available Embedded analytics Very low Low

    Strategic Roadmap

    Phase 1 - MVP (Ship immediately)

    1. Meta Graph API - Lead Ads + page posting
    2. Zapier - Triggers + actions + REST hook
    3. Make - Same API as Zapier

    Phase 2 - Core (First 90 days)

    1. GA4 - Dashboard widgets, campaign analytics
    2. LinkedIn - Social publishing, lead-gen forms
    3. Mixpanel - Product usage scoring, cohort sync

    Phase 3 - Niche / Future (6+ months)

    1. Twitter/X - Posting, listening (requires paid tier)
    2. Tableau - Only if demand; prioritize Power BI
